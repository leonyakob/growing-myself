## 1. 使用边界

1. 本文件只服务 Stage 5 的后续 QA，不改产品阅读材料，不回写 `/home/king/github/growing-myself/路遥/人生/《人生》阅读笔记.md`。

## 3. 验证办法

### 3.1 Happy checks

1. `read(filePath="/home/king/github/growing-myself/.omo/evidence/ren-sheng-stage5-question-compression/task-05-regression-sentinels.md")`
   - 确认表中至少有 12 行哨兵，且包含七个 router 哨兵，`ID 003`、`ID 006`、`ID 021`、`ID 109`、`ID 117`、`刘玉海救灾处`、`黄亚萍的物质付出`。
   - 确认表中另外覆盖 Stage 5 风险，高加林过度道德化、巧珍主动性被擦掉、黄亚萍真实成本被压平、德顺只剩 moral mouthpiece、跨作品联动变堆名字。
   - 确认每一行都含有 `risk`、`source`、`failure pattern`、`required Stage 5 check`、`where later task should apply it` 五栏。
2. 人工抽读表格
   - 确认每一行都是条件触发的 QA 检查，而不是“每题必须提到某人物、某材料”的内容配额。
   - 确认凡是 `source` 只来自 B 级导航索引、C 级规则或计划要求的行，都明确要求后续回到 A 级原卡。
3. `git status --short --untracked-files=all`
   - 对照 `task-01-source-ledger.md:171-210` 的 dirty worktree 基线解释，只把 evidence 目录下的增量视为允许变化。
   - `.omo/evidence/ren-sheng-stage5-question-compression/` 下若出现并行 sibling artifacts，也按允许的 evidence 目录变化处理，不误判成产品文件污染。
   - `/home/king/github/growing-myself/路遥/人生/《人生》阅读笔记.md`、两份 prompt 文件、`AGENTS.md`、plan、ledger 若有新增改动，则判失败。

### 3.2 Failure checks

1. 若任何一行写成“每个最终问题都必须提到……”或等价意思，立即改写成条件触发的 QA 哨兵。
2. 若任何一行只给出 B 级导航、C 级规则或计划来源，却没有写明“后续必须回 A 级卡”，立即补写，否则该哨兵无效。
3. 若任何一行把 external-only、高赞材料或跨作品联动写成可直接替代《人生》本书证据，立即改写。
4. 若 `git status --short --untracked-files=all` 显示本次变化扩散到 evidence 目录外，或落到产品阅读材料、规则文件、计划文件，立即判失败并回退。
5. 若哨兵语气把人物压扁成道德标签，而不是保护人物复杂性，也算失败，必须重写风险描述。
