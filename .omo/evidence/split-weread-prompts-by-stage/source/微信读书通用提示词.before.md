# 微信读书 Skill 通用整理提示词

用途：当阅读任意一本书时，用这份通用提示词生成该书的专用微信读书整理提示词。专用提示词应固定作者、书名、中间整理稿路径和正式阅读笔记路径，避免每次整理时混用通用规则和具体书目规则。

核心目标：

1. 用微信读书数据完整保留我的阅读现场：划线、想法、颜色、章节、位置，以及同位置其他书友高赞想法。
2. 先生成中间整理稿，再优化分析，最后迁移到正式阅读笔记。
3. 中间整理稿是临时工作台，正式阅读笔记才是长期保留文档。
4. 所有分析以文本为落点，以文学性和分析深度为目标，不为了覆盖数量牺牲质量。
5. 正式阅读笔记必须保留可读证据层：书籍划线原文、我自己写的原想法，以及确有保留价值的外部读者原话。AI评价、AI修正、AI补充只能建立在这些原文之上，不能替代它们。
6. `source ID`、`bookId`、`chapterUid`、`range`、点赞数、抓取状态和官方版定位等技术字段，只服务中间整理、迁移清单和回源核对，不默认进入正式阅读笔记正文。

---

## 0. 占位说明与路径规则

- `{作者}`：书籍作者或项目中的作者目录名，例如 `路遥`、`余华`。
- `{书名}`：书名，例如 `人生`、`活着`。
- `{中间稿路径}`：建议为 `/home/king/github/growing-myself/{作者}/{书名}/《{书名}》中间整理稿.md`。
- `{正式笔记路径}`：建议为 `/home/king/github/growing-myself/{作者}/{书名}/《{书名}》阅读笔记.md`。
- `{专用提示词路径}`：建议为 `/home/king/github/growing-myself/{作者}/{书名}/《{书名}》微信读书提示词.md`。
- `{个人笔记bookId}`：我的划线、颜色和个人想法所在 bookId。导入书、云书或外部文件通常会有独立 bookId。
- `{官方公开版bookId}`：微信读书官方书城版或公开热门划线所在 bookId。若暂未确认，在专用提示词中写“待确认”。
- `{bookId关系}`：同一 bookId / 导入版与官方版不同 / 待确认。

不要把中间整理稿命名成单纯的 `{书名}.md`，避免和书目文件、正式笔记混淆。

### 0.1 不可降级原则与任务入口门

开始任何整理、优化、迁移、回看或提示词修改前，先执行以下门槛：

1. 先阅读当前任务适用的 `AGENTS.md`，再判断任务类型：第一阶段、第二阶段、第三阶段、第四阶段（全书收束整合）、只整理、只评价、只咨询、提示词修改或状态检查。
2. 质量、文本证据、用户原意和来源区分，高于速度、上下文经济、格式统一和覆盖数字。
3. 分批只改变执行组织方式，不能遗漏已纳入范围的用户材料，不能删减证据或降低应有的单卡深度。这不等于保留所有外部读者候选，也不等于把所有轻卡升级为完整卡。
4. 划线无法回源、bookId 映射不明、接口失败或资料不完整时，必须保留真实状态，不得猜测、补写或伪造对应材料。
5. 上下文不足时，停在可验证的批次边界，说明已完成范围、待确认材料和下次起点。不要压缩证据或以总论冒充完成。
6. 复杂任务先过计划门：先起草计划，再交给 Momus、Metis、Oracle 审查，吸收审查意见后把审查版计划呈现给用户。只有用户明确授权执行，才进入执行阶段。
7. 用户平时不需要手动切换到 Prometheus 或 Atlas；若任务进入计划流或执行流，可由 Hephaestus 按当前运行时、当前客户端已注册的入口接续。
8. 只读咨询、规则解释和状态检查，不触发文件编辑、Git 操作或任何执行入口调用。

## 1. 如何用通用提示词生成某本书专用提示词

```text
请根据 `/home/king/github/growing-myself/微信读书通用提示词.md`，为《{书名}》生成一份专用微信读书整理提示词。

要求：
1. 把所有 `{作者}` 替换成实际作者目录名。
2. 把所有 `{书名}` 替换成实际书名。
3. 固定中间整理稿路径为 `/home/king/github/growing-myself/{作者}/{书名}/《{书名}》中间整理稿.md`。
4. 固定正式阅读笔记路径为 `/home/king/github/growing-myself/{作者}/{书名}/《{书名}》阅读笔记.md`。
5. 专用提示词里不要再保留通用占位符。
6. 专用提示词要能直接复制使用。
7. 写入 `/home/king/github/growing-myself/{作者}/{书名}/《{书名}》微信读书提示词.md`。
8. 生成专用提示词时，应确认该书是否存在“个人笔记 bookId”和“官方公开版 bookId”不同的情况；若已知，应在专用提示词中固定写出两个 bookId；若未知，应保留“待确认”字段，并要求第一轮整理前先确认公开热门划线使用的官方版 bookId。
```

### 专用提示词生成边界与验收门

1. 只迁移可跨书复用的方法、流程门和质量标准，不迁移其他书目的具体人物、主题结论、bookId、源材料编号、历史误读或回归样本。
2. 书目专属事实未经核实，一律标为“待确认”，不能借用其他书的资料补全。
3. 初次生成专用提示词不得虚构回归样本。只有该书出现真实、可定位的问题后，才可在该书专用提示词中沉淀相应的回归哨兵。
4. 生成后检查：专用提示词不保留通用占位符，路径明确，未知事实已标明“待确认”，且没有混入其他书目的专属内容。

## 2. 四阶段总流程

整理微信读书材料时分四阶段：

1. **第一阶段：生成中间整理稿**
   抓取本轮新增划线、想法和同位置其他书友高赞想法，合并、排序、轻量预分类，写入 `{中间稿路径}`。第一阶段只做整理、轻量预分类和高赞想法初步筛选，不做深度评价，不写入正式阅读笔记。
   - 进入条件：用户明确要求整理本轮新增材料，或明确要求执行第一阶段。
   - 输入材料：适用的 `AGENTS.md`、本提示词、微信读书划线与我的想法数据、可稳定定位时的外部读者材料，以及已有中间整理稿的最新游标。
   - 退出条件：本轮材料按阅读顺序入稿，划线原文与我的原想法均未遗漏，卡片只完成轻量预分类；已核对本轮范围、匹配待确认项和下次接着整理位置。
2. **第二阶段：优化中间整理稿**
   在中间整理稿内，对本轮新增内容进行轻卡优化、完整卡优化、主卡候选深挖、暂存素材归类。第二阶段重点是分析质量和文学表达，不为了覆盖所有材料而牺牲深度。
   - 进入条件：第一阶段材料已完成，并通过本阶段应有的 QA。
   - 输入材料：适用的 `AGENTS.md`、本提示词、本轮中间整理稿中的划线原文、我的原想法和已保留的外部读者原话。
   - 退出条件：本轮纳入优化范围的材料均有与卡片类型相称的处理，完整卡和主卡通过单卡质量门，轻卡仍然轻而有味，我的原想法保持原样。
3. **第三阶段：迁移到正式阅读笔记**
   把已经优化完成、适合沉淀的内容分类安置到 `{正式笔记路径}`。正式阅读笔记中不保留中间稿编号、临时流程说明和冗余归档痕迹。
   - 进入条件：用户明确要求更新正式阅读笔记，第二阶段优化完成并通过 QA。
   - 输入材料：已优化的中间整理稿、迁移前回源预检清单、正式阅读笔记现有结构，以及本提示词第 11 节的迁移规则。
   - 退出条件：已通过迁移前回源预检、迁移后防回归 QA 和最终验证。材料有明确去向，正式稿保留可读证据锚点而不保留技术定位痕迹。
4. **第四阶段：读完整本书后的全书收束整合**
   在整本书读完、第三阶段已有基础后，以 **阅读现场档案 + 文章素材索引** 对全书材料做一次整书级收束。第四阶段重点是补齐全书结构、保留阅读轨迹、建立文章方向导航，不把文章素材索引写成文章草稿，也不把第三阶段全部重跑一遍。
   - 进入条件：整本书已经读完，且用户明确要求执行全书收束整合。
   - 输入材料：正式阅读笔记现有结构、必要时回源的已优化中间稿或用户原句、仅供内部使用的整合台账，以及本提示词第 12 节的规则。
   - 退出条件：正式稿形成可读的阅读现场档案与文章素材索引，阅读轨迹与判断变化有去处，正式稿正文不保留技术定位痕迹。

未经明确要求，不要直接把微信读书导出的内容写入正式阅读笔记。

## 3. 多轮整理规则

一本书通常会分多轮整理。中间整理稿不按轮次新建多个文件，而是持续追加到同一个 `《{书名}》中间整理稿.md`。

每一轮用独立小节，例如：

```markdown
## 第1轮整理：开头至第六章

## 第2轮整理：第七章至第十章

## 第3轮整理：……
```

规则：

1. 第 N 轮第一阶段，只追加第 N 轮新增内容，不重写前几轮内容。
2. 第 N 轮第二阶段，默认只优化第 N 轮新增内容，不重新优化整个中间稿。
3. 第 N 轮第三阶段，默认优先迁移第 N 轮已优化完成、适合沉淀的内容；同时检查前几轮已经优化但尚未迁移到正式阅读笔记的遗留内容。若这些遗留内容仍有价值，应一并分类安置；若暂时不迁移，应在中间稿中标明“待迁移 / 待回看”，避免长期遗漏。
4. 只有当我明确说“回看全书中间稿”“整合前几轮”“重整全部内容”时，才跨轮重写或整体重排。
5. 每轮结束都要给出“下次接着整理位置”，方便下一轮继续。

## 4. 阶段切换前 QA 与条件式 Git 检查点

当我要求进入第二阶段时，例如：

```text
现在根据【提示词】进行第二阶段，优化《{书名}》中间整理稿
```

AI 必须先完成第一阶段的 QA：

1. 检查第一阶段生成的 `{中间稿路径}`。
2. 核对本轮范围、划线原文、我的原想法、匹配待确认项和下次接着整理位置。
3. 确认第一阶段没有深度评价，也没有改写正式阅读笔记。

当我要求进入第三阶段时，例如：

```text
现在根据【提示词】进行第三阶段，迁移到《{书名}》正式阅读笔记
```

AI 必须先完成第二阶段的 QA：

1. 检查第二阶段优化后的 `{中间稿路径}`。
2. 核对本轮优化范围、卡片类型处理、原想法保留和核心卡质量门。
3. 确认第二阶段没有改写正式阅读笔记。

注意：

- QA 是阶段切换的必要条件，阶段切换本身不构成 commit 或 push 授权。
- 只有当前用户指令明确要求 commit、push 或其他 Git 操作时，才执行相应操作。当前用户指令优先于本提示词的一般规则。
- 获得明确 Git 授权后，先查看 `git status`，只处理本阶段相关文件，不混入无关阅读材料、临时文件或其他书目的内容。
- 仍需遵守项目 `AGENTS.md` 的 Git 规则。如项目要求 Git 命令加前缀，则使用 `GIT_MASTER=1`。

## 5. 数据抓取硬约束

微信读书里“划线颜色”和“我的想法”来自不同数据口径：

- 划线原文、章节、位置、颜色：来自 `/book/bookmarklist`，颜色字段是 `colorStyle`。
- 我的个人想法：来自 `/review/list/mine`。

### 5.0 中间整理稿与正式阅读笔记的数据边界

中间整理稿是可回源的工作台，应保留 `source ID`、`bookId`、`chapterUid`、`range`、匹配方式、抓取状态和官方版定位等技术字段，供合并、核对、迁移清单和 QA 使用。

正式阅读笔记是面向阅读的长期文档，只保留划线原文、我自己写的原想法和必要的外部读者原话等可读证据锚点。技术字段不能替代划线原文，不默认写入正文，也不能以标题或编号形式伪装成证据层。

整理微信读书笔记时，默认以“我的想法”为主索引整理，优先处理有我的想法的条目；只有划线、没有我的想法的条目暂不整理，除非我明确要求整理纯划线。

这里的“以我的想法为主索引”，不是只整理我的想法文本。AI 仍必须同时获取两路数据，用划线接口补足原文、颜色、章节和位置：

- 划线原文、章节、位置、颜色：来自 `/book/bookmarklist`，颜色字段是 `colorStyle`。
- 我的个人想法：来自 `/review/list/mine`。

优先用：

```text
bookId + chapterUid + range
```

合并同一段原文。

不要只输出我的想法，也不要只靠想法接口整理；必须尽量合并出对应的划线原文和划线颜色。

同时，若某条“我的想法”已经能稳定合并到划线原文，应默认尝试抓取同位置其他书友高赞想法，作为外部读者材料。这里的“同位置”优先指同一段文本位置，而不是个人笔记所在 `bookId + chapterUid + range` 必须和官方公开版完全相同。默认抓取只围绕“我写了想法的位置”，不是扫描全章、全书或所有热门划线。

### 5.1 跨段划线与想法匹配规则

不要因为 bookmarklist 和 review/list/mine 的 range 不完全一致，就直接判断为“只有想法”或“无法匹配”。跨段划线、段尾想法、第二段想法都可能导致 range 不完全相同。

匹配顺序：

1. **精确匹配**：bookId + chapterUid + range 完全一致。
2. **跨段匹配**：bookId 和 chapterUid 相同，想法 range 落在某条跨段划线 range 内。
3. **重叠匹配**：bookId 和 chapterUid 相同，想法 range 与划线 range 有重叠或相邻，且文本内容连续。
4. **文本判断匹配**：想法内容明显对应某条划线原文，即使 range 不完全一致，也可合并，但必须标注“非精确 range 匹配”。

输出时增加：

- 匹配方式：精确 range / 跨段 range / 重叠 range / 文本判断匹配 / 未匹配
- 是否需要人工确认：是 / 否

效率要求：

- 不要为了少量未精确匹配条目反复重拉全书、重整旧材料。
- 未匹配但有想法的条目先列入“有想法但划线匹配待确认”，不要自动扩大整理范围。
- 只有划线、没有想法的条目，本轮不整理，除非我明确要求整理纯划线。

每条整理材料至少保留：

- 章节名
- `chapterUid`
- `range`
- 匹配方式
- 是否需要人工确认
- 划线颜色
- 划线原文
- 我的原想法
- 同位置其他书友高赞想法抓取状态：已获取并保留 / 已获取但未保留 / 未能定位官方版同位置 / 同位置暂无公开想法 / 接口错误
- 官方版定位：bookId / chapterUid / range（若可定位）
- 同位置其他书友高赞想法摘录：按卡片价值保留 0-5 条
- 与我的想法的初步关系：回声 / 补充 / 挑战 / 反向解释 / 独立精彩 / 暂无有效高赞想法
- 想法类别
- 卡片判断
- 后续处理方向

如果接口为了定位“本轮新增”需要先拉取全书划线或全部想法，只允许在内部匹配；输出时只展示本轮新增内容，不要把已整理过的旧材料重新输出。

如果某条想法没有 `range`，或按以上匹配规则仍无法和划线原文稳定合并，应单独放入“有想法但划线匹配待确认”小节，不要强行归并，也不要因此重整旧材料。

### 5.2 同位置其他书友高赞想法默认伴随抓取规则

整理本轮新增内容时，默认以“我写了想法的位置”为索引，尝试查询同一文本位置下其他书友的公开高赞想法。

术语定义：

- **同位置其他书友高赞想法**：指我在某段文本 A 上划线并写下个人想法后，其他书友也在同一段文本 A 或高度重叠文本上发表的公开想法中，点赞数相对靠前，且能对文本理解形成回声、补充、挑战、反向解释或独立启发的内容。
- **同位置**：优先按文本内容对应判断，不按个人笔记所在 `bookId + chapterUid + range` 直接相同判断。若个人笔记来自导入书、云书或外部文件，应先确认并映射到微信读书官方公开版坐标。
- **高赞**：以 `/book/readreviews` 返回的 `likesCount` 为主要发现信号，但不以点赞数作为唯一收录标准。同质重复的高赞想法只保留 1 条代表；方向不同或互相冲突且都贴文本的高赞想法，应同时保留。

bookId 判断规则：

1. 若个人笔记 bookId 和官方公开版 bookId 相同，可优先使用同一 `bookId + chapterUid + range` 尝试定位。
2. 若个人笔记来自导入书、云书或外部文件，个人划线和想法可能挂在导入版 bookId 上，而公开热门划线和其他书友高赞想法可能挂在官方书城版 bookId 上。
3. 不得因为个人笔记 bookId 下 `/book/bestbookmarks` 为空，或个人 range 直接调用 `/book/readreviews` 返回错误，就判定同位置没有高赞想法。
4. 专用提示词中应记录：个人笔记 bookId、官方公开版 bookId、bookId 关系、官方版定位方式。

接口分工：

1. `/book/bookmarklist`：抓个人划线原文、章节、range、颜色 `colorStyle`。
2. `/review/list/mine`：抓我的个人想法、abstract、chapterUid、range。
3. `/book/bestbookmarks`：抓官方版热门划线，含官方版 range、划线文本和划线人数；用于文本匹配和公开版定位，但每章最多返回 20 条，不覆盖全部位置。
4. `/book/underlines`：抓官方版章节内划线热度 range、count、score，但不含文本；只能作为辅助信号，不能单独证明文本同位置。
5. `/book/readreviews`：用官方版 `bookId + chapterUid + range` 抓该划线下其他书友公开想法；核心字段包括 `review.content`、`review.abstract`、`review.range`、`likesCount`、`commentsCount`、`repostCount`。
6. `/review/list`：是整本书公开点评 / 公开想法列表，不能按我的章节和 range 定位同一划线位置，不作为同位置高赞想法主接口。

第一阶段内部必须先做轻量预分类：

1. 先抓取并合并我的划线、颜色、个人想法。
2. 只根据划线原文和我的原想法，轻量判断为：轻卡 / 完整卡候选 / 主卡候选 / 暂存素材。
3. 轻量预分类只为整理调度和高赞想法抓取深度服务，不等同于第二阶段深度评价。
4. 根据预分类决定同位置高赞想法候选池深度。

同位置匹配优先级：

1. **精确文本匹配**：我的划线原文和官方版热门划线原文完全一致。
2. **包含匹配**：我的划线原文是官方版划线的一部分，或官方版划线是我的划线的一部分。
3. **高重叠匹配**：两段文字大部分重叠，只是前后多了一两句。
4. **章节内近似文本匹配**：章节标题相同，核心句子相同，但 range、段落边界不同。
5. **无法稳定匹配**：找不到足够相同的公开版文本，才标“未能定位官方版同位置”。不能因为我的个人 range 没出现在热门划线列表里，就直接判定失败。

候选池与保留数量：

1. 所有能稳定定位官方版同位置的条目，默认至少尝试抓取一批 `/book/readreviews` 公开想法，通常 `count=20`。
2. 轻卡默认保留 0-1 条；只有当一条外部高赞想法能明显照亮我的触动、形成挑战或补充时才保留。
3. 完整卡候选默认保留 1-3 条；如果高赞想法出现多个不同解释方向，应优先保留差异，而不是只保留点赞最高的一条。
4. 主卡候选 / 文章核心候选默认保留 2-5 条；必要时可继续翻页扩大候选池，例如最多查看 40 条，但必须服务于解释分歧、主题张力或问题链推进。
5. 保留多条时必须说明每条分工：回声 / 补充 / 挑战 / 反向解释 / 独立精彩。
6. 同质重复的外部高赞想法只保留 1 条代表；方向相反但都贴文本的外部高赞想法应优先并存。

筛选规则：

1. 按 `likesCount` 发现候选，但不要机械取最高赞。
2. 先过滤纯骂人、纯表情、纯感叹、复读原文、鸡汤化、名言堆砌但不贴文本、离文本太远、只做道德审判但不能推进理解的内容。
3. 再按观点方向分簇：回声、补充、挑战、反向解释、独立精彩、暂不采用。
4. 每个观点簇里优先选点赞较高、贴文本、表达清楚、能推进我的问题链的想法。
5. 若同一划线下出现多条高赞想法，且方向相反但都能贴文本解释，应标为“外部读者分歧”，不要只保留更赞同我的一条。

输出状态必须区分：

1. **已获取并保留**：已定位官方版同位置，并筛出有效高赞想法。
2. **已获取但未保留**：接口有外部想法，但多为低质量、同质重复或不能推进文本分析。
3. **未能定位官方版同位置**：个人笔记坐标与官方公开版坐标不同，当前未能通过原文匹配找到对应官方版 range。
4. **同位置暂无公开想法**：已定位官方版同位置并调用 `/book/readreviews`，但 totalCount 为 0。
5. **接口错误**：调用失败、鉴权失败、服务端错误等，应记录错误码或简要原因。

高赞想法只能作为外部读者材料，不得替代我的感受、问题和判断。

## 6. 颜色默认流向

微信读书划线颜色通过 `colorStyle` 字段识别。颜色只作为默认流向，不是最终卡片类型。

内部识别规则：

- `colorStyle=1`：粉/红色，轻卡，记录气氛、心动、暂不深挖的触动。
- `colorStyle=2`：紫色，人物卡，记录人物心理、关系、命运选择。
- `colorStyle=3`：蓝色，主题卡，记录阶层、城乡、贫穷、劳动、尊严、现实结构。
- `colorStyle=4`：青色，细读卡，记录写法、场景、动作、物件、空间、结构、沉默。
- `colorStyle=5`：黄色，对照卡，记录跨作品、同书前后、作者生平、现实经验联动。
- `colorStyle=0`：未分类旧材料。

输出时不要把颜色写成单纯数字。正文中写成：

```markdown
- 划线颜色：蓝色（主题卡默认流向）
```

不要写成：

```markdown
- colorStyle：3
```

整理时必须保持阅读顺序，不要为了按颜色分类而打散阅读线索。颜色统计和主卡候选可以放在文末汇总。

## 7. 想法类别与卡片类型

### 7.1 想法类别

“想法类别”指我在微信读书里写下的想法本身是什么形态。

1. **感受型**：情绪、心动、惊讶、喜欢、心疼、厌恶、共鸣等，可能很短，但要保留。
2. **浅判断型**：已经有判断，但证据、问题链、文本分析还不充分。
3. **问题型**：主要提出疑问，暂时没有充分判断。
4. **片段分析型**：已经开始分析人物、情节、动作或主题，但还不完整。
5. **完整分析型**：已经接近正式阅读笔记里的完整写法，包含“触动点 / 为什么停下来 / 问题 / 判断 / 扩写方向”等要素，或虽然没显式分栏但分析已经比较完整。
6. **素材型**：主要是为以后文章保存证据、句子、对照材料，当前不急着分析。

### 7.2 卡片类型

“卡片类型”指这条材料后续怎样处理：

- **轻卡**：保留触动、感受、气氛和直觉，不强行深挖。轻卡不是废料，它记录真实阅读心流。
- **完整卡候选**：有明显人物心理、动作细节、结构反常、主题张力，可以补成完整阅读卡。
- **主卡候选**：不仅能补成完整卡，而且可能发展成文章核心、章节核心或全书母题。
- **暂存素材**：目前主要作为证据、对照材料或主题材料保存，暂不做完整评价。

轻卡和完整卡不是按字数机械区分，而是看它是否已经形成可追问的矛盾和证据链。

## 8. 文学分析质量规则

第二阶段优化时，评价重点放在分析质量、分析深度和文学表达上。

最高原则：**AI 不是润色器，而是细读推进器。** AI 的任务不是把我的判断写得更漂亮，而是通过细读文本，主动发现我问题链中最有文学潜力、最容易被忽略的解释分支；必要时要反驳我的判断，也要反驳 AI 自己先前的顺手判断。

开始完整卡或主卡候选分析前，必须先判断：我的原问题里最有价值、最锋利、最不能被总论覆盖的分叉是什么。不能用作品外的大主题、习惯性概括或漂亮总论压过具体词语、动作、沉默和心理雾区。

### 8.1 证据锚点与文本落点

完整卡和主卡候选的分析必须回到具体文本细节，例如：

- 词语
- 动作
- 表情
- 沉默
- 场景
- 人物关系
- 叙事位置
- 前后情节的变化

不能只用“阶层”“自尊”“爱情”“城乡差异”“命运”等抽象词概括。

完整卡、主卡候选和正式阅读笔记中的核心卡，必须至少绑定一个可回到原文的证据锚点，可以是划线原句，也可以是清楚可辨的词语、动作、表情、场景、沉默、人物关系或叙事位置。轻卡只需保留一个简短文本落点或轻判断，不必承担完整卡的分析负担。

文学作品与非虚构作品都适用这一原则，但分析重点不同：文学作品侧重词语、动作、人物心理和叙事结构；非虚构作品侧重概念、论证、证据、结构、修辞和隐含前提。不得把小说人物心理分析硬套到非虚构作品，也不得把非虚构的概念概括替代文学作品的细读。

### 8.2 人物心理要写出过程，不要直接贴标签

分析人物时，要写清楚心理动作链：

- 人物意识到了什么；
- 他回避了什么；
- 他想维护什么；
- 他害怕承认什么；
- 他在哪一点上发生动摇；
- 他还没有走到哪一步。

不要直接把人物心理写成没铺垫的结论。

例如，不要直接写：

```text
他已经被对方的爱打动了。
```

应写出中间层次：

```text
这里更值得分析的不是人物是否已经“动情”，而是他已经察觉到对方的情感，并本能地回避这种可能。对方的真诚先触到的可能不是爱情本身，而是他的身份想象、自尊防线或关系压力：他不愿承认自己会被一个“不符合自我理想位置”的人影响。
```

具体判断仍需根据实际文本调整，不要套用示例。

### 8.3 文本歧义与心理雾区不能一笔带过

遇到“不知为什么”“有时”“最好”“似乎”“仿佛”“老半天”“开玩笑”“笑了笑”“沉默”等模糊词、力度词、停顿和沉默时，不能直接越过。这些词常常是人物心理复杂性的入口。

处理这类细节时，至少先辨析：

1. 这个词修饰的是哪一个念头、动作或情绪；
2. 它的力度是确定、犹豫、试探、退缩，还是自我遮掩；
3. 它打开了哪些可能解释，这些解释是否互斥，还是可以同时存在；
4. 当前文本最多能支持哪一层判断，哪一层需要后文验证。

但不要机械逐词阐释。只有当模糊词、力度词、停顿、沉默和人物选择、关系转折、叙事留白或主题张力有关时，才作为心理雾区处理。分析必须说明：为什么这个词值得停下。

### 8.4 完整卡和主卡候选必须打开矛盾

每张完整卡或主卡候选至少要指出一个张力点。这里的“张力”不是固定主题清单，而是要求 AI 从当前文本里寻找相反力量的碰撞，例如：

- 人物想要什么 vs 现实不允许什么；
- 人物说出口的话 vs 没说出口的沉默；
- 表面动作 vs 隐藏心理；
- 情感冲动 vs 身份、伦理、习俗或制度限制；
- 自我想象 vs 他人眼光；
- 个人选择 vs 时代、阶层、家庭或命运结构；
- 叙事明写的内容 vs 叙事刻意留白的内容；
- 一个场景里的温情、羞耻、恐惧、欲望、尊严等复杂力量之间的冲突。

这些只是寻找张力的方法，不是要求每本书都套用同一组主题。分析必须从当前作品、当前章节和当前文本细节出发，找到真正存在的矛盾，不要因为示例里出现某类张力就强行套用。

### 8.5 AI 要提供表达增量

AI 不能只说：

```text
这个判断有道理。
这个问题很敏锐。
可以继续发展。
```

必须把用户没说透的地方说出来：

- 补心理层次；
- 补文本细节；
- 补问题链；
- 补文章方向；
- 补一个新的合理视角；
- 尽量用有文气但不空泛的语言表达。

所谓“表达增量”，不是替用户改成学术论文，而是在保留用户真实反应的基础上，把直觉说深，把判断说细，把矛盾说出文学味。

### 8.6 AI 可以反驳，但必须指出缺失的分析环节

如果用户判断跳跃、太宽、太二选一、证据不足或表达不准，AI 应直接指出。

但不要只说：

```text
这不准确。
这里需要修正。
这个判断太跳。
```

而要说明：

- 少了哪一环；
- 哪个文本细节还没接上；
- 当前判断最多能推出什么；
- 如果要推进成更强判断，需要补什么证据或中间分析；
- 可以怎样改成更稳、更深、更有文学性的表达。

### 8.7 大胆分析，但不要省掉过程

可以借助后文回看、作者背景、社会结构、同书内部联动和跨作品比较提出深层解释。

但不能直接空降结论。要写出为什么能这样看、从哪里推进过来、当前文本能支撑到哪一步。

深度不是保守，深度是把判断的来路写出来。

### 8.8 升级问题链后必须给出阶段性回答

完整卡、主卡候选和正式阅读笔记中的核心卡，不能只停在“问题链升级”。如果已经提出了较强的问题链，AI 必须继续补“当前回答（沿原问题）”“当前回答（新增角度）”和必要的“后文待回看”。

问题链升级必须分两层：

1. **问题链升级（沿原问题）**：保留我的核心关切，把问题问得更准、更贴文本。
2. **另一条升级问题链（新增角度）**：从当前文本真实存在的另一条线索或写法角度重新打开问题，不预设固定主题清单，不把某本书的主题强行套到其他作品里。

这里的回答不是标准答案，也不是把后文封死，而是说明：根据当前文本细节，问题目前最多能回答到哪里，哪些判断已有证据，哪些还需要后文验证。

推荐写法：

```markdown
- 问题链升级（沿原问题）：
- 另一条升级问题链（新增角度）：
- 当前回答（沿原问题）：
- 当前回答（新增角度）：
- 后文待回看：
```

要求：

1. 回答必须落到具体文本细节，不能只用抽象概念概括。
2. 如果有两条问题链，就分别回答；不能把“沿原问题”和“新增角度”压成一个漂亮总论。
3. 如果用户的问题包含“是 A，还是 B，还是 C”，必须逐项回应：A/B/C 各自的文本依据、限制、是否可共存，以及当前文本更稳的阶段性判断。
4. 回答要写成阶段性判断，不要写成盖棺定论。
5. 如果当前证据不足，要明确写“只能回答到这里”，并指出后文需要验证什么。
6. 轻卡不强行补回答；只有完整卡、主卡候选、文章核心候选需要补。
7. 原则是：问题链负责把思考打开，阶段性回答负责把思考落到文本，后文待回看负责保留弹性。

当原问题是多分支问题时，推荐展开为：

```markdown
- 当前回答（沿原问题）：
  - A 的文本依据与限制：
  - B 的文本依据与限制：
  - C 的文本依据与限制：
  - 当前更稳的判断：
```

### 8.9 AI修正必须深化，不只是整理语序

AI修正不是简单整理语序，也不是把我的话改得更顺。它必须在保留原意的基础上完成至少一个深化动作：

1. 补出我没有说透的心理机制；
2. 补出文本细节和判断之间的桥梁；
3. 补出人物动作背后的身份、关系、处境或叙事压力；
4. 把原判断从“我觉得”推进到“文本如何让这个判断成立”。

如果 AI修正只是复述、压缩、顺句或换同义词，视为不合格，必须重写。

AI修正还必须独立可读：离开前后栏目也能看出它在修正哪一段材料、依据哪个细节、得出什么判断。不要只写“这里”“上面所说”“这种情绪”等无法回指的句子。

### 8.10 文学表达不是轻卡专属

文学味儿不只用于轻卡。完整阅读卡、人物主线卡、主卡候选和文章核心候选，也必须在分析准确的基础上，适当给出有文学表达力的句子。

但文学表达必须服从文本和分析，不能为了漂亮而失真。

合格的文学表达应满足：

1. 落到具体动作、物件、场景、沉默或人物关系；
2. 能推进分析，而不是只制造修辞；
3. 不编造文本没有的情绪；
4. 不把人物写得比文本更极端；
5. 能作为正式阅读笔记中的句子使用。

每张完整卡 / 主卡候选至少补一处“可入笔记的文学表达”，可以是一句，也可以是一小段。

### 8.11 分析后命名，而非先命名硬套

高质量书评不是先制造漂亮标题，再硬把文本塞进去；而是先细读行为链、心理链和上下文，等分析成熟后，如果自然出现一个能统摄全段的表达，再把它前置为书评开头。

要求：

1. 命名是分析完成后的压缩结果，不是分析开始前的模板。
2. 如果没有自然生成的统摄表达，不要硬造标题。
3. 像“三板斧”这类表达，只有在后文行为链、心理链和上下文都能支撑时才成立。
4. 不要把某个成功书评标题固化成通用套路；方法可以沉淀，命名不能机械套用。

## 9. 第一阶段：生成中间整理稿


### 10.2 轻卡优化规则

轻卡适用于感受型、浅判断型、短问题、气氛记录。处理目标是“轻而有味”：保留真实触动和用户原句的生命力，同时把表达稍微修亮，不把它硬拽成完整论文，也不压成无味摘要。

每张轻卡按以下结构补在原想法下面：

```markdown
**AI轻评：**

- 这条想法抓住了什么：
- 可以稍微说深的一点：
- 不必强行展开的原因：

**AI优化表达：**

> 
```

要求：

- 先保护真实触动，不强行拔高。
- 至少保留触动点、原想法中有生命力的一句，以及一个简短文本落点或轻判断。
- 如果我的表达太直白，可以优化成更有文学味儿的表达。
- 可以引用相关名言或典故，但必须准确、贴切；不能为了好看硬套，不能编造出处。不确定出处时，不要写成“某某说”。
- 优化表达直接写在原想法下方，不覆盖原想法。
- 不强行补“我提出的问题 / 我的初步判断 / 以后可扩写成什么文章”。
- 不强行补完整问题链、阶段性回答或跨作品联动。轻卡略丰满不是问题，只有明显写成完整卡、破坏整体层次时才减重。

### 10.5 同位置其他书友高赞想法差异分析规则

第二阶段处理同位置其他书友高赞想法时，重点不是判断“谁说得更好”，而是看外部读者材料怎样照亮我的阅读位置。

每条有效高赞想法按关系分簇处理：

1. **外部读者回声**：与我的想法同向，能证明这个触动点不是孤立的。
2. **外部读者补充**：补出我没有说到的细节、人物心理、叙事安排或时代背景。
3. **外部读者挑战**：与我的判断不同，但能贴文本提出另一种解释。
4. **反向解释**：不是简单反驳，而是从相反立场重读同一细节，使原问题更复杂。
5. **独立精彩外部视角**：与我的原想法没有直接关系，但本身贴文本、有张力、有问题意识或有表达穿透力，能够独立推进作品理解。

判断规则：

- 点赞数、热度或接口默认排序只负责发现，不负责最终定性。
- AI 必须根据文本质量判定高赞想法是否值得保留：是否贴住具体细节，是否打开人物心理、叙事安排或主题张力，是否能形成可追问的问题，是否避免鸡汤化和道德化。
- 与我的想法无关但独立精彩的高赞想法，不要强行解释成“对我的想法的补充”；直接标为“独立精彩外部视角”。
- 方向相反但都贴文本的高赞想法，应并存为“外部读者分歧”，不要只保留更赞同我的一条。
- 外部读者原话必须逐字保留，优先随对应卡片内联，并明确标为“外部读者原话”及其关系：回声、补充、挑战、反向解释或独立精彩。不得改写成用户第一人称判断，也不得概括为“有读者说”。
- 只有无法归入具体卡片、却仍独立精彩的外部材料，才进入全局“外部读者精彩想法”小节。外部读者材料不是每张卡的必填项。
- 第二阶段可以在对应条目下保留“同位置其他书友高赞想法分析”，但不得让高赞想法覆盖我的原想法、AI评价、AI修正和AI补充。

### 11.3 正式阅读笔记游标更新规则

每次更新正式阅读笔记时，文末只保留一条最新的“下次接着整理位置”。

1. 先删除第 N 轮旧的“下次接着整理位置”。
2. 再迁移第 N+1 轮内容，并处理前几轮已优化但尚未迁移的遗留内容。
3. 第 N+1 轮迁移完成后，在文末追加新的“下次接着整理位置”。

不要在正式阅读笔记中累积多条历史游标。

### 11.4 所有材料最终都要有归宿

中间整理稿只是临时工作台。整本书读完后，所有有价值材料都应被分类安置：

- 主卡候选 → 重点卡 / 文章核心候选；

- 轻卡 → 轻卡存档 / 阅读心流片段；

- 无法定位想法 → 散点想法 / 待回看 / 不迁移。

第三阶段不是只迁移“精华”，而是把不同价值层级的材料放到合适位置。

### 11.5 外部读者材料迁移规则

第三阶段迁移同位置其他书友高赞想法时，优先随对应卡片内联保存。只有无法归入具体卡片、却仍能独立推进作品理解的材料，才进入正式阅读笔记中的全局“外部读者精彩想法”小节。不要把独立精彩的想法单独归入主题素材库或人物素材库。

“同位置其他书友高赞想法”内部区分五种来源关系：

1. **外部读者回声**：与我的想法形成同向关系。
2. **外部读者补充**：补出我没展开的文本细节、心理机制或叙事安排。
3. **外部读者挑战**：提出与我的判断不同但有文本依据的解释。
4. **反向解释**：从相反立场重读同一细节，使问题更复杂。
5. **独立精彩外部视角**：与我的想法无直接关系，但高赞想法本身足够贴文本、有张力、有问题意识或有表达穿透力。

迁移要求：

- 第五类只是不形成回声，不代表价值低；如果精彩，应和前四类一起随卡内联或在必要时进入全局小节。
- 不要为了归档方便，把第五类硬塞进我的原判断，也不要改写成“我的想法的补充”。
- 外部读者原话必须按原话保存，不得概括成“有读者说”，不得改写为用户第一人称判断。可以在原话后附一两句 AI 简评，说明它精彩在哪里、打开了哪个文本角度。
- 外部读者材料不是每张卡的必填项。与我的想法没有直接关系而独立精彩的材料，不强行解释，仍可进入全局小节。
- 点赞数、热度或接口默认排序只作为发现信号；正式迁移仍以文学分析质量为准。

### 11.6 迁移后防回归 QA

迁移完成后，逐项完成以下 QA，未通过不得把迁移视为完成：

1. 本轮每个稳定条目编号都有正式稿去向、待回看去向或明确的不迁移理由。
2. 每个已迁移条目的划线原文都已进入正式稿，我自己写的原想法没有被 AI 改写覆盖。
3. 保留的外部读者材料仍是原话，优先随卡内联，没有被概括为“有读者说”或改写为用户判断。
4. 正式稿正文没有 `source ID`、`bookId`、`chapterUid`、`range`、点赞数、抓取状态、官方版定位或过程性编号等工作台痕迹。
5. AI修正能独立读懂，有明确文本落点，不依赖“这里”“上面”等模糊回指。
6. 轻卡仍然轻而有味，没有被拔高成完整卡，也没有被压成无味提纲。
7. 已迁移材料和迁移前回源预检清单逐条核对，没有遗漏、错位或无去向材料。

## 12. 第四阶段：读完整本书后的全书收束整合

只有当我明确说“读完整本书后做全书收束整合”“全书收束整合”“按全书整理正式阅读笔记”之类的话时，才执行这一阶段。

第四阶段的标准模型是 **阅读现场档案 + 文章素材索引**。核心原则是：**卡片档案为底座，文章素材索引为上层导航**。

### 12.1 第三阶段与第四阶段的边界

- 第三阶段处理的是“本轮或前几轮已优化材料如何迁入正式阅读笔记”。
- 第四阶段处理的是“整本书读完后，如何把已经有归宿的卡片、修订链、待回看项和文章方向收束成整书结构”。
- 第三阶段的任务重心是迁移优化后的轮次材料。第四阶段的任务重心是全书收束整合。
- 第四阶段不是把第三阶段全局重跑一遍。除非材料缺失、重复、语义错误、判断已被后文修正，或被明确标记为待重迁，否则不要整体重迁已有正式稿。
- 若正式稿已有合格卡片，默认补结构、补链接、补阅读轨迹，不重复搬运。
- 第四阶段要保留阅读过程，不把早期浅判断、误读、犹豫和后来的修正偷偷磨平。

### 12.2 产物边界：内部整合台账与正式阅读笔记

- **内部整合台账** 只供整合、判重、回源和 QA 使用，可以出现 `source ID`、`range`、重复判定、`source status`、`archive disposition`、处理动作、来源版本、`QA state`、是否 external-only 等技术字段。
- **正式阅读笔记** 只保留读者可读的归档卡片和紧凑的文章导航，不写整行台账字段，不贴技术状态标签。
- 正式阅读笔记正文里的锚点必须是可读锚点，比如卡片标题、章节或场景名、引文短句、我的原句片段。`source ID`、`range` 不能作为读者可见的文章方向锚点，也不能出现在正式阅读笔记正文。
- 原始 `我自己写的内容` 必须保留，AI修正只能另写，不能覆盖原文。
- 外部读者原话若保留，必须保持原话身份，不能改写成我的判断。

### 12.3 多轴台账，不用单一“最终状态”替代材料

全书收束时，先给每条材料建立多轴判断，不要只贴一个“完成/未完成”标签：

| 轴 | 允许值与用法 |
|---|---|
| `source status` | `已在正式笔记中，结构合格，只补文章索引` / `已在正式笔记中，但需补 readable evidence / 用户原句 / 外部原话` / `中间稿已优化，尚未迁移` / `中间稿与正式稿重复，需合并或建立修订链` / `正式稿旧判断已被后文修正，保留为阅读轨迹` / `仅外部读者材料，无用户卡片锚点` / `来源不足，待回看` |
| `archive disposition` | `保留为轻卡` / `保留为完整卡` / `提升为核心卡` / `合并入其他卡` / `归档不迁移` / `待回看` |
| `evidence role` | `主题证据` / `人物线证据` / `写法证据` / `阅读轨迹证据` |
| `article links` | 0 个、1 个或多个文章方向 |
| `trajectory flags` | `早期误读` / `后文修正` / `判断变化` |
| `external-reader relation` | `回声` / `补充` / `挑战` / `反向解释` / `独立精彩` / `丢弃并说明理由` |

处理顺序也按这个台账走：先判 `source status`，再判 `archive disposition`，再判 `evidence role`、`article links`、`trajectory flags` 和 `external-reader relation`。不要跳过前两步，直接把材料压成“文章素材”或“最终结论”。

### 12.4 一卡双归宿，与两层正式稿结构

`一卡双归宿` 的意思是，每张有价值的卡都先保留自己的档案身份，再决定是否被 0 个、1 个或多个文章方向引用。它可以只留在档案里，也可以同时服务多个写作方向。

两层结构如下：

1. **阅读现场档案**，负责保留或补齐卡片本体、修订链、必要的外部原话、轻卡身份、完整卡判断和核心卡层级。
2. **文章素材索引**，负责把可写方向、核心问题、关联卡片、原始触动、张力和证据缺口排成可导航结构。**文章素材索引只引用卡片，不吞掉卡片**。

文章素材索引里的常用引用形态：

```markdown
- 关联卡片：卡片标题 / 章节或场景 / 引文短句
- 原始触动：保留我的原句或原句片段
- 证据作用：它在这个文章方向中承担什么功能
- 缺口：还缺哪类文本证据或对照材料
```

### 12.5 全书收束后的正式阅读笔记骨架

```markdown
## 全书收束整合

### 一、阅读现场档案
> 保留或补齐卡片本体；已存在且合格的卡片不重复搬运。

### 二、文章素材索引
> 只做导航，不吞掉卡片，不写成文章草稿。

| 文章方向 | 核心问题 | 关联卡片 readable anchor | 原始触动/用户原句 | 张力 | 可用证据 | 缺失证据 |
|---|---|---|---|---|---|---|

### 三、阅读轨迹与判断变化
> 只收束真正能说明阅读变化的误读、犹豫、修正，不机械罗列全部过程。

### 四、待回看 / 归档不迁移
> 简短记录原因；详细处理台账留在内部证据，不进入正式稿正文。
```

正式阅读笔记正文里，`阅读现场档案` 必须在 `文章素材索引` 前面。正式稿只保留可读结构，不出现 `source status`、`archive disposition`、`source ID`、`range`、抓取状态或 QA 状态。

### 12.6 判重、回源与修订链规则

判重时先看来源关系，再看解读关系。

- 来源关系看四类：`同一引文` / `引文重叠` / `同一场景` / `不同引文`。
- 解读关系看五类：`同一判断` / `修正判断` / `不同角度` / `外部回声` / `外部挑战`。

处理原则：

1. `同一引文` 或 `引文重叠`，且属于 `同一判断`，选更清楚的一张做代表卡，另一张记入内部整合台账，`archive disposition` 记为 `合并入其他卡`。
2. `同一引文`，但属于 `修正判断`，不要硬并成一张最终卡，而要建立修订链。当前可用判断进 `阅读现场档案`，较早读法进 `阅读轨迹与判断变化`。
3. `同一场景` 但 `source ID` 或 `range` 不同，只要动作链和核心张力相同，就视为同一场景卡处理。正式稿使用可读场景名和引文短句，不用技术来源号做锚点。
4. `不同引文` 且属于 `不同角度`，只要承担的 `evidence role` 不同，就分别保留，不为了整齐硬合并。
5. `外部回声` 和 `外部挑战` 只能附着或照亮用户卡片，不能顶替用户卡片成为正式稿的主体。

### 12.7 误读修正与阅读轨迹写法

当旧判断被后文修正时，不删旧卡，按以下结构收束：

- `当时的读法`
- `后来出现的证据`
- `全书后的修正`
- `误读的价值`
- `文章索引用法`

普通文章方向默认引用“全书后的修正”。只有当文章本身讨论阅读过程、判断如何变化、我如何被叙事带偏时，才把误读链作为 `阅读轨迹证据` 一并挂入文章素材索引。

### 12.8 轻卡、外部读者材料与防膨胀规则

- 轻卡即使被文章素材索引引用，仍然可以保持轻卡，不自动升级为完整卡。它可以承担气氛证据、语言风格证据、主题回声或文章开头材料。
- 只有外部读者材料、没有用户卡片锚点的条目，先记入内部整合台账，`source status` 记为 `仅外部读者材料，无用户卡片锚点`。这类材料不能直接改写成我的第一人称判断，也不能直接充当文章索引的主锚点。
- 外部读者原话可以是 `回声`、`补充`、`挑战`、`反向解释` 或 `独立精彩`，但都不能覆盖我的原判断，不能把外部评论改写成“我其实一直这样想”。
- 鸡汤化、重复感叹、脱离文本太远的外部材料，可以记为 `丢弃并说明理由`，不要为了显得丰富硬塞进正式稿。
- 不是每张卡都必须进入正文卡片区。`归档不迁移` 和 `待回看` 都是有效去向。
- 为了防止正文膨胀，文章素材索引只写紧凑导航，不复制整张卡正文，不堆同义弱卡，不把内部判重理由整行贴进正式阅读笔记。
- 原始 `我自己写的内容` 不能被 AI修正覆盖。即便全书后判断改变，也是在下方补修正，不是回头抹掉当时的原句。

### 12.9 全书收束整合 QA

第四阶段完成前，必须对照 `.omo/evidence/fixtures/add-whole-book-consolidation-model.md` 做 dry-run。至少要覆盖：部分迁移卡、同引文同判断重复、同引文改判、同场景不同 `source ID/range`、早期误读后期修正、外部高赞挑战、external-only 强评论、被文章引用但不升级的轻卡、以及防正文膨胀的 `归档不迁移` case。

QA 未通过时，不得把全书收束整合视为完成。至少逐项确认：

1. 正式稿里先有 `阅读现场档案`，后有 `文章素材索引`。
2. `文章素材索引只引用卡片，不吞掉卡片`，没有把整张卡正文复制成文章提纲。
3. 正式稿正文和文章方向锚点里没有 `source ID`、`range` 等技术字段。
4. 轻卡没有因为被文章引用就被强行升级成完整卡。
5. 外部读者原话没有被改写成我的判断。
6. `我自己写的内容` 没有被 AI修正覆盖。
7. 有价值但不进正文的材料，已经在 `待回看 / 归档不迁移` 或内部整合台账中有明确去向。

## 13. 大任务处理策略

如果本轮内容体系庞大，不要为了完成任务牺牲分析深度。复杂任务只改变执行组织方式，不遗漏已纳入本轮范围的用户材料，也不降低单卡深度。

### 13.1 正常任务

符合以下情况时，可以直接执行：

- 处理 1-2 张完整卡 / 主卡候选；
- 或主要处理轻卡、暂存素材；
- 或完整卡 / 主卡数量虽不多，且主题线比较集中；
- 或只是抓取、合并、轻量优化。

正常任务仍必须遵守“文学分析质量规则”。不能因为数量少，就只做润色、概括或格式填充。

### 13.2 复杂任务

符合以下任一情况时，必须先制定计划，再分步执行：

- 至少 1 张核心卡牵涉多条人物线、主题线、叙事结构、后文回看或跨作品联动；
- 涉及多个目标文件、多个阶段产出或正式阅读笔记结构调整；
- 人物线、主题线、叙事结构、现实背景、写法分析等多条线索交织；
- 需要把多轮中间稿整合成章节化、主题化的正式笔记；
- 用户明确强调不要牺牲深度和质量。

复杂任务的执行原则：

1. 先判断任务范围和最需要深挖的卡片。
2. 再按主卡候选、完整卡候选、轻卡、暂存素材，或按章节分批处理。
3. 每一批都必须逐条回到划线原文、我的原想法和具体文学分析。
4. 分批不是删减；计划不是替代细读。
5. **不遗漏已纳入本轮范围的用户材料；外部读者候选仍按价值筛选，不以数量保留为目标；不降低单卡深度，只改变执行组织方式。**

复杂任务的计划、审查和执行授权是三个独立的门：

1. 先起草计划，写清任务范围、分批策略、质量门、涉及文件和预期产出。
2. 起草后必须交给 Momus、Metis、Oracle 审查，不能跳过其中任何一方。
3. 必须把三方审查意见吸收进计划，补齐范围、风险、QA 和执行边界，再向用户呈现审查后的计划。
4. 用户批准计划，只代表认可这份审查后的方案，不等于已经明确授权执行。
5. 用户平时不需要手动切换到 Prometheus 或 Atlas；若当前运行时由 Hephaestus 负责路由，应由它把计划流和执行流接到对应工作流。
6. “开始执行计划”或 `start work` 可以视为自然语言执行意图，但仍必须映射到当前客户端、当前运行时已注册的 start-work 入口。
7. 具体入口写法要按当前环境说明执行。用户所在客户端里，可能是 `/start-work <plan-name>`；技能文档示例里，也可能是 `$start-work <plan-name>`。这两个都只是环境示例，不能被宣称为所有客户端的唯一通用写法。
8. 不要把裸写 `start-work` 当成面向用户的通用执行命令；若当前环境没有可用的已注册 start-work 入口，必须暂停并说明，不能静默直接执行。
9. 用户明确说“审查补全后执行”时，可视为用户提前给出“审查完成后再执行”的意图，但仍必须先完成 Momus、Metis、Oracle 审查、吸收审查意见，并在满足当前客户端入口规则后再执行。

### 13.3 止损与定点校补

正式稿已达标后，只做定点校补：围绕用户指出的卡片、证据、结构或表述修正，不借机重写整章或全书。只有用户明确要求全书重整时，才重新制定计划，按 13.2 完成审查，确认范围并取得明确执行授权。

## 14. 辅助提示词快捷任务模板

以下片段只是快捷任务模板，也就是 shortcut task templates，方便快速表达需求，不是更高优先级的工作流规则。

使用这些模板时，仍必须同时服从：`AGENTS.md`、第 0.1 节任务入口门、第 11 节迁移与 QA 要求、第 12 节全书收束整合规则、第 13 节复杂任务计划门，以及复杂任务必须先经过 Momus、Metis、Oracle 审查、吸收审查意见、取得明确执行授权后才能进入执行的规则。

这些模板不能绕过 QA、Git 授权、执行授权，也不能绕过当前运行时、当前客户端已注册的 start-work 入口。若模板内容与第 11 节迁移前回源预检或迁移后防回归 QA 重复，以第 11 节为准；若与第 12 节全书收束整合规则重复，以第 12 节为准。

### 只整理，不评价

## 15. 专用提示词回归样本沉淀规则

每本书只在出现真实、可定位的 AI 误读、漏读、过度概括、磨平用户文气或迁移错位风险后，才在该书的专用提示词中沉淀“回归样本”或回归哨兵。

回归样本应写明：触发场景、错误类型、必须保留或核对的证据、禁止重犯的处理方式和可验证的通过条件。它服务于该书的后续整理与 QA，不应被复制到通用提示词或其他书目的专用提示词。

通用提示词不虚构任何书目的回归样本。初次生成专用提示词时，只保留本节的机制，不填入具体案例。
