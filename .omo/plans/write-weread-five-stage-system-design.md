## B. 当前系统理解

### B1. 系统形成脉络

根据已读上下文，本系统不是从“工具效率”出发，而是从长期文学阅读训练中被逼出来的边界系统。形成链条可以概括为：

1. **文学素养训练目标**：项目根定位是“作家驱动的主题化阅读”，目标是保留用户真实、有文气的反应，同时让分析更深、更准、更贴文本。
2. **微信读书实验材料**：微信读书保存划线、想法、颜色、章节位置、同位置其他书友高赞想法，能保存阅读现场，但材料天然碎片化。
3. **早期补规则阶段**：为防止用户原话被改写、外部读者材料错位、轻卡被拔高、事实漂移、复杂任务偷跑，逐步补了许多规则。
4. **阶段重构阶段**：补规则不够，因为不同认知动作在同一任务里冲突。系统改为五阶段，给 AI 不同阶段不同认知权限。
5. **第五阶段写作转化阶段**：《人生》的第五阶段验证了：第四阶段只保住材料结构还不够，第五阶段必须把整书材料压成核心问题、反证、张力和写作路径。
6. **QA / 回归样本 / plan gate 语义保护**：QA 不是格式检查，而是防止文本、材料身份和任务授权发生语义漂移。

### B2. 五阶段的核心认知权限

正式设计文档必须把“五阶段”写成认知权限架构，而不是单纯流程清单：

| 阶段 | 认知权限 | 不允许越界 |
|---|---|---|
| 第一阶段：生成中间整理稿 | 保存权、合并权、轻量预分类权 | 没有深评权；不得写正式阅读笔记 |
| 第二阶段：优化中间整理稿 | 分析权、问题升级权、判断修正权 | 没有正式归档权；不得覆盖用户原想法 |
| 第三阶段：迁移到正式阅读笔记 | 归档权、回源预检权、来源身份固定权 | 没有覆盖原文权；不得做全书收束 |
| 第四阶段：全书收束整合 | 全书结构收束权、文章素材导航权 | 没有吞掉卡片权；不得把索引写成文章 |
| 第五阶段：全书回看与写作转化 | 问题压缩权、反证修正权、写作转化权 | 没有重跑整理权；不得直接替代最终成文 |

### B3. 系统的关键分层

正式设计文档必须让维护者能分清以下层级：

1. **根规则层**：`AGENTS.md`，定义项目定位、文学分析工具箱、卡片边界、四段式归档、微信读书处理边界和 Git 规则。
2. **通用 router 层**：`微信读书通用提示词.md`，负责五阶段入口、通用边界、通用 QA、计划门、新书生成规则。
3. **通用阶段正文层**：五个通用阶段提示词文件，负责各阶段执行正文、输入输出、禁止事项和 QA。
4. **书目专用 router 层**：例如《人生》router，负责固定路径、作者书名、个人 bookId、官方公开版 bookId、书目事实、书目回归样本和专用阶段入口。
5. **书目专用阶段正文层**：例如《人生》第四、第五阶段文件，承接通用方法，但写入书目专属事实、锚点、问题链和回归哨兵。
6. **材料产物层**：中间整理稿、正式阅读笔记、第四阶段全书收束成果、第五阶段写作转化成果。
7. **计划与 QA 工件层**：`.omo/plans`、必要时 `.omo/drafts`、`.omo/evidence`、回归样本 / 回归哨兵、`.omo/run-continuation` 等。

### C1. 根规则

已读取：`/home/king/github/growing-myself/AGENTS.md`

关键吸收点：

- 项目定位：长期阅读、思考和写作训练，采用“作家驱动的主题化阅读”。
- 训练结构：用作家保证深度，用主题训练思想，用作品提供证据，用背景增加纵深，用对照防止狭窄，用写作完成转化。
- 反馈原则：中文读书笔记用中文回复；直接、具体、诚实；先保留有生命力的句子，再补分析；文气必须有文本落点。
- 核心工具箱：五异法扫描、问题阶梯、张力地图。
- 卡片规范：轻卡不强行深挖；主卡必须围绕核心问题展开证据、判断、问题链和文章方向；细节卡从具体往抽象走；主题卡从抽象往具体落。

- 外部读者材料规则：同位置其他书友高赞想法只能作为外部读者材料，不能替代用户自己的感受、问题和判断。
- 归档四段式：我自己写的内容、AI评价、AI修正、AI补充；AI 修正不能覆盖用户原文。
- 文件与 Git 规则：只在用户要求时修改项目文件；未经要求不碰其他阅读材料；Git 操作必须单独授权。

关键吸收点：

## E. 正式文档定位与写作策略

### E1. 正式文档应该是

- 项目内部设计说明；
- 长期维护文档；
- 可执行、可检查、可扩展；
- 给未来 session 和维护者看的系统说明；
- 指导新书接入、阶段判断、QA、回归样本维护和复杂任务计划门的稳定文档。

### E2. 正式文档不应该是

- 方法论散文；
- 文学长文；
- 项目宣传文；
- 提示词全文复制；
- 更新日志；
- 《人生》专用说明书；
- README 式泛泛介绍；
- 技术流程堆砌。

### E3. 写作策略

1. **结构优先**：章节按系统维护问题排列，而不是按方法论文章的情绪 / 论证节奏排列。
2. **表格优先**：文件结构、五阶段工作流、风险、防护、QA 层级、新书接入，均优先用表格或清单。
3. **规则句明确**：必须 / 不得 / 只有 / 若未知 / 待确认等边界语要写清。
4. **案例克制**：《人生》只在需要说明专用层、第四阶段、第五阶段、回归样本时短用，不承担全文叙事主轴。
5. **不复制提示词正文**：正式文档只提炼系统设计，不复制阶段提示词里的执行模板、长段 QA、完整例子。
6. **避免系统自嗨**：明确系统的代价、风险和防护，不写成“这套系统很强”的宣传说明。

## G. 各章关键内容展开计划

### G1. 第 1 章：系统定位

必须写清：

1. 这套系统不是自动笔记整理器；
2. 它服务于长期文学阅读、思考和写作训练；
3. 核心目标是保留真实阅读现场，同时推进分析、归档、全书收束和写作转化；
4. 最终目标不是整齐笔记，而是文学素养提升；
5. 根项目定位是“作家驱动的主题化阅读”：
   - 用作家保证深度；
   - 用主题训练思想；
   - 用作品提供证据；
   - 用背景增加纵深；
   - 用对照防止狭窄；
   - 用写作完成转化。

### G2. 第 2 章：核心原则

至少列出并简短解释 10 条原则：

1. 文本优先；
2. 用户原始感受优先保留；
3. 文气是优势，但必须有文本落点；
4. 提问可以跳远，论证不能偷步；
5. 轻卡不强行深挖；
6. AI 修正不能覆盖用户原文；
7. 外部读者材料不能伪装成用户判断；
8. 文章索引不能吞掉卡片；
9. 第五阶段不回退成整理；
10. 通用层和专用层不得互相污染。

每条原则都要可用于 QA，不要写成格言。

### G3. 第 3 章：文件结构总览

必须使用表格：

```markdown
| 文件 / 目录 | 作用 | 是否产品文件 | 维护注意 |
|---|---|---|---|
```

至少覆盖：

- `AGENTS.md`；
- `微信读书通用提示词.md`；
- 五个通用阶段提示词文件；
- 书目专用 router；
- 书目专用阶段提示词文件；
- 中间整理稿；
- 正式阅读笔记；
- 第四阶段全书收束成果；
- 第五阶段写作转化成果；
- `.omo/plans`；
- `.omo/drafts`；
- `.omo/run-continuation`；
- 回归样本 / 回归哨兵。

必须明确产品文件 / 计划工件 / 草稿工件 / 运行续跑工件之间的区别。

### G4. 第 4 章：五阶段工作流

必须使用表格：

```markdown
| 阶段 | 核心认知动作 | 输入 | 输出 | AI 权限 | 禁止事项 | QA |
|---|---|---|---|---|---|---|
```

每阶段必须包含：

1. 第一阶段：生成中间整理稿；保存现场；只有保存权，没有深评权。
2. 第二阶段：优化中间整理稿；升级问题和判断；有分析权，没有正式归档权。
3. 第三阶段：迁移到正式阅读笔记；长期归档；有归档权，没有覆盖原文权。
4. 第四阶段：全书收束整合；全书结构收束；有全书收束权，没有吞掉卡片权。
5. 第五阶段：全书回看与写作转化；有写作转化权，没有重跑整理权。

### G5. 第 5 章：通用层与书目专用层边界

必须说明：

通用层负责：

- 通用阶段方法；
- 通用 QA；
- 通用计划门；
- 新书生成规则；
- 通用边界。

书目专用层负责：

- 固定路径；
- 作者 / 书名；
- 个人 bookId；
- 官方公开版 bookId；
- 书目专属阶段入口；
- 书目事实；
- 书目回归样本；
- 书目专属问题和锚点。

必须强调：

- 通用层不能写入《人生》三问、人物、九个锚点；
- 专用层不能把本书经验倒灌成通用模板；
- 新书生成专用提示词时不得借用其他书目的事实、bookId、回归样本或专属判断。

### G6. 第 6 章：阅读现场保护机制

必须说明：

- 微信读书划线保存原文锚点；
- 想法保存第一反应；
- 颜色保存用户粗分类；
- 高赞想法只能作为外部读者材料；
- 中间整理稿是临时工作台；
- 正式阅读笔记才是长期档案；
- 正式归档采用四段式：
  1. 我自己写的内容；
  2. AI评价；
  3. AI修正；
  4. AI补充。
- AI 修正不能覆盖用户原文；
- 外部读者材料不能伪装成用户判断。

还要说明卡片层级：

- 轻卡只保留气氛、心动或暂不深挖的触动；
- 完整卡才系统评价；
- 主卡 / 核心卡才承担文章方向；
- 细节卡从具体往抽象走；
- 主题卡从抽象往具体落。

### G7. 第 7 章：写作转化机制

必须说明第五阶段进入条件：

- 整本书已经读完；
- 第四阶段已经完成；
- 已有阅读现场档案和文章素材索引；
- 用户明确要求进入第五阶段。

第五阶段必须包含：

1. 核心问题压缩；
2. 证据-反证表；
3. 反证如何修正判断；
4. 张力地图；
5. 五异法细读点；
6. 写作转化路径；
7. 作家写法档案；
8. 跨作品联动。

必须强调：

- 核心问题不是主题词；
- 反证不是装饰；
- 张力地图必须落回文本；
- 五异法不硬凑；
- 写作方向必须包含“最容易写坏的地方”；
- 作家写法档案必须区分“本书观察”和“后续待验证假设”。

### G8. 第 8 章：QA 与回归样本

必须说明 QA 的目标不是格式检查，而是语义保护。

建议使用三层表：

| QA 层级 | 防护目标 | 典型错误 | 典型检查 |
|---|---|---|---|
| 文本层 | 防事实、措辞、引文意义漂移 | “晚到了五个小时”被改成“在路上耽误五个小时” | 回源核对、引文意义核对 |
| 材料层 | 防用户原话、AI 补充、外部读者想法混淆 | 外部高赞想法改写成用户判断 | 四段式归档、外部原话身份检查 |
| 任务层 | 防阶段误判、计划变执行、复杂任务越权修改 | `.omo/plans` 被当成执行授权 | plan gate、审批门、Git 授权检查 |

回归样本规则必须写清：

- 只从真实错误中沉淀；
- 不虚构回归样本；
- 每个回归样本写明触发场景、错误类型、必须保留的证据、禁止重犯的处理方式和通过条件；
- 回归样本属于书目专用层，不复制到其他书；
- 通用层只保留机制，不写具体书目样本。

### G9. 第 9 章：复杂任务计划门与 `.omo/plans` 规则

必须完整纳入 `.omo/plans` 规则。建议按以下小节写：

1. **何时触发计划门**：复杂任务、阶段计划门、提示词修改计划门、用户明确要求建 plan。
2. **计划必须写什么**：目标、范围、输入、输出、风险、目标文件、验收标准、禁止事项。
3. **三方审查职责**：
   - Metis：查隐藏意图和真实需求；
   - Momus：查计划结构、遗漏和可执行性；
   - Oracle：查架构风险和长期维护风险。
4. **审批门**：用户批准计划不等于执行授权，除非用户明确说批准执行或开始执行。
5. **`.omo/plans` 默认规则**：触发计划门且无上位 no-write 禁令时，可直接落盘计划工件，不需额外询问。
6. **上位禁止覆盖**：若用户明确“不写任何文件 / read-only / 不落盘计划 / 不写 `.omo`”，不得写 `.omo/plans` 或 `.omo/drafts`，除非用户明确豁免。
7. **`.omo/drafts` 限制**：不是每次必需，只在用户要求或 workflow / skill 明确需要 durable draft 时使用。
8. **非执行边界**：`.omo/plans` / `.omo/drafts` 不等于产品文件修改授权。
9. **变更报告**：区分计划 / 草稿工件与产品目标文件。
10. **Git 边界**：Git 操作必须单独授权。

### G10. 第 10 章：新书接入流程

必须写成可执行 checklist：

1. 建立作者 / 书名目录；
2. 生成书目专用 router；
3. 生成五个书目专用阶段提示词；
4. 固定中间整理稿路径；
5. 固定正式阅读笔记路径；
6. 固定专用 router 和阶段文件路径；
7. 确认个人 bookId；
8. 确认官方公开版 bookId；
9. 若未知，标注待确认，不得编造；
10. 第一阶段前确认公开热门划线映射；
11. 不虚构回归样本；
12. 后续出现真实问题后再沉淀书目回归哨兵。

### G11. 第 11 章：维护与扩展规则

必须说明：

- 修改通用提示词时，不得混入书目专属事实；
- 修改书目专用提示词时，不得污染通用层；
- 阶段文件只写本阶段执行正文；
- router 只写入口、路径、规则摘要，不复制阶段长正文；
- 回归样本要定期检查是否仍有效；
- 第五阶段专用经验要先经具体书目验证，再考虑是否抽象进通用层；
- 如果规则越来越厚，应优先检查是否需要拆阶段、拆文件、拆边界，而不是继续堆补丁。

### G12. 第 12 章：已知风险与防护

必须用表格：

```markdown
| 风险 | 表现 | 防护方式 |
|---|---|---|
```

至少包括：

1. 提示词过厚；
2. 阶段误判；
3. 轻卡被过度加工；
4. 用户原文被 AI 修正覆盖；
5. 外部读者想法混入用户判断；

## I. `.omo/plans` 规则纳入方案

正式设计文档第 9 章必须完整纳入 `.omo/plans` 规则。写作原则如下：

1. **放置位置**：第 9 章“复杂任务计划门与 `.omo/plans` 规则”。
2. **表述方式**：用清单和审查者职责表，不写成散文化解释。
3. **核心句必须出现**：`.omo/plans` 是计划工件，不是产品文件修改授权。
4. **默认落盘规则**：当任务触发复杂任务计划门、阶段计划门、提示词修改计划门，或用户 / 当前适用提示词明确要求建 plan 时，AI 可直接创建或更新 `.omo/plans` 下的计划工件，不需要额外询问是否允许写 `.omo/plans`。
5. **上位禁止规则**：若用户当轮明确“不写任何文件 / 不要修改任何文件 / read-only / do not edit files / 只在聊天里给计划 / 不落盘计划 / 不创建计划文件 / 不写 `.omo`”，不得创建或更新 `.omo/plans` 或 `.omo/drafts`，除非用户明确豁免。
6. **`.omo/drafts` 限制**：只在用户要求、workflow / skill 明确要求 durable draft，或确实需要 durable 草稿时使用。
7. **非执行边界**：写计划工件不允许据此修改提示词、阅读笔记、中间稿、成果文件、文章草稿或其他产品文件。
8. **变更报告规则**：报告时必须把计划 / 草稿工件与产品目标文件分开。
9. **Git 规则**：Git 操作必须另行授权；阶段切换、计划批准、计划文件存在都不等于 commit 授权。

防误解写法：

- 不写“计划批准后即可执行”；要写“用户批准计划只代表认可方案，不等于执行授权，除非用户明确说批准执行或开始执行”。

- 不把 `.omo/drafts` 写成每次必需工件。
