# 微信读书通用提示词优化计划（待执行）

## 0. 状态

- 目标文件：`/home/king/github/growing-myself/微信读书通用提示词.md`
- 参考文件：`/home/king/github/growing-myself/路遥/人生/《人生》微信读书提示词.md`
- 本计划只修改通用提示词，不修改《人生》专用提示词、阅读笔记、中间稿、AGENTS.md 或其他书目文件。
- 本次修改的目标是把《人生》最终提示词中可跨书复用的流程护栏上升为通用规则，同时避免把《人生》人物、主题、bookId、源 ID 或定点样本写进通用提示词。

## 1. 必改问题

1. 通用提示词目前缺少“不可降级原则与任务入口门”。
2. 复杂任务规则没有准确写出 `/shared/start-work`，容易把 `start-work` 当普通短语。
3. 三阶段流程缺少进入条件、输入材料和退出条件。
4. Git 检查点可能被误解为阶段切换天然授权提交/推送。
5. 正式阅读笔记迁移规则只强调四段式，容易漏掉书籍划线原文。
6. 中间稿技术字段和正式稿读者正文的边界不够清楚。
7. 轻卡规则还不是“轻而有味”，容易过度压缩或拔高。
8. 外部读者材料仍偏向统一归入全局小节，容易切断它与具体卡片的关系。
9. 缺少迁移前去向清单和迁移后防回归 QA。
10. 缺少每本书专用提示词沉淀本书回归哨兵的通用机制。

## 2. 不做的事

1. 不复制《人生》的具体人物、主题判断、bookId、源 ID、定点回归样本。
2. 不把所有书都当小说；保留文学作品与非虚构作品的分析分支。
3. 不要求每张卡都成为完整卡或主卡。
4. 不要求每张卡都有跨作品联动。
5. 不要求每张卡都有外部读者材料。
6. 不要求正式阅读笔记显示逐段 source ID、bookId、chapterUid、range、点赞数或抓取状态。
7. 不自动提交或推送。

## 4. 具体规则

### 4.1 核心目标

新增两条：

1. 正式阅读笔记必须保留可读证据层：书籍划线原文、我的原想法、确有保留价值的外部读者原话。
2. source ID、bookId、chapterUid、range、点赞数、抓取状态和官方版定位只服务中间整理、迁移清单和回源核对，不默认进入正式正文。

### 4.2 专用提示词生成边界

在第 1 节新增“专用提示词生成边界与验收门”：

- 只迁移通用方法、流程门和质量标准。
- 不迁移其他书目的具体人物、主题结论、bookId、源材料编号、历史误读或回归样本。
- 书目专属事实未核实时写“待确认”。
- 初次生成专用提示词不得虚构回归样本。

### 4.3 不可降级原则与任务入口门

新增任务入口门：

- 先读适用的 AGENTS.md。
- 先判断任务类型。
- 质量、文本证据、用户原意和来源区分高于速度、上下文经济、格式统一。
- 分批不得遗漏纳入范围的用户材料；这不等于保留全部外部读者候选，也不等于升级全部轻卡。
- 划线无法回源、bookId 映射不明、接口失败或资料不完整时，必须保留真实状态，不得猜测或补写。
- 上下文不足时停在可验证批次边界，不通过压缩证据或总论冒充完成。
- 只读咨询、规则解释和状态检查不得触发文件编辑、Git 操作或 `/shared/start-work`。

### 4.4 三阶段流程

为第一、第二、第三阶段分别补：进入条件、输入材料、退出条件。

### 4.5 Git 检查点

将第 4 节改为“阶段切换前 QA 与条件式 Git 检查点”：

- QA 必做。
- commit/push 只有在当前用户指令明确要求时执行。
- 当前用户指令优先于通用提示词中的一般规则。

### 4.6 数据边界

新增“中间稿与正式稿的数据边界”：

- 中间稿保留技术定位字段。
- 正式稿只保留可读证据锚点。
- 技术字段不能替代划线原文。

### 4.7 文学分析质量规则

修改：

- “必须有文本落点”改为“证据锚点与文本落点”。
- 完整卡、主卡、正式核心卡至少绑定一个原文证据锚点。
- AI修正必须独立可读，不能依赖“这里、上面、这种情绪”等模糊回指。
- 文学作品与非虚构作品都适用：文学作品侧重词语、动作、人物心理、叙事结构；非虚构作品侧重概念、论证、证据、结构、修辞和隐含前提。

### 4.8 轻卡

改为“轻而有味”：

- 保留真实触动、用户原句生命力、简短文本落点或轻判断。
- 不强行补完整问题链、阶段性回答、文章方向或跨作品联动。
- 轻卡略丰满不是问题，只有明显变成完整卡、破坏层次时才减重。

### 4.9 外部读者材料

改为：

- 外部读者原话随对应卡片内联优先。
- 必须标为“外部读者原话”并标明关系。
- 不得放入“我自己写的内容”，不得改写成用户第一人称判断，不得概括成“有读者说”。
- 只有无法归卡但独立精彩的材料才进入全局“外部读者精彩想法”小节。

### 4.10 第三阶段迁移

新增迁移前回源预检清单：

| 稳定条目编号 | 划线原文 | 我的原想法 | 外部读者原话 | 正式稿去向 | 处理动作 |
|---|---|---|---|---|---|

正式卡结构：

```markdown
**【划线原文】**
> 原书划线原文。

**【我自己写的内容】**
> 我的原想法。

**【AI评价】**

**【AI修正】**

**【AI补充】**

**【外部读者原话｜关系：回声 / 补充 / 挑战 / 反向解释 / 独立精彩】**（如有）
> 外部读者原话。

- AI简评：
```

新增迁移后防回归 QA。

### 4.11 复杂任务与 `/shared/start-work`

准确写法：

- `/shared/start-work` 是准确技能名。
- `start work` 或“开始执行计划”可作为自然语言触发语。
- `start-work` 只作为易混简称，不单独作为执行命令。
- `/start work` 不是准确命令写法。
- 计划确认和授权执行是两个门；只有用户明确授权执行时才调用 `/shared/start-work`。
- 如果用户在同一条指令中明确要求“审查补全后执行”，可视为计划补全后执行授权，但仍必须实际调用 `/shared/start-work`。
- 若环境没有 `/shared/start-work`，必须暂停说明，不能静默直接执行。

### 4.12 止损规则

正式稿达标后只做定点校补；用户明确要求全书重整时，仍需重新计划、确认范围、再执行。

### 4.13 专用提示词回归样本沉淀规则

新增通用机制：每本书出现真实、可定位的 AI 误读、漏读、过度概括、磨平用户文气等风险后，才在该书专用提示词沉淀回归哨兵。通用提示词不虚构样本。
