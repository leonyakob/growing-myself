## 1. 任务入口规则

开始执行前，先判断这次请求是否属于第一阶段。

属于本阶段的常见入口：

1. 用户明确要求“整理本轮新增材料”“生成中间整理稿”“做第一阶段”。
2. 用户要求“只整理，不评价”。
3. 用户要求“只判断轻卡、完整卡、主卡候选”，且目标仍是把材料先放进中间整理稿。
4. 用户要求围绕“我写了想法的位置”补查同位置其他书友高赞想法，但仍以中间稿整理为目标。

不属于本阶段的入口：

1. 只咨询规则、只做状态检查、只改提示词。
2. 要求深度评价、优化表达、补问题链、补阶段性回答。
3. 要求把内容更新到正式阅读笔记。
4. 要求读完整本书后的全书收束整合。

执行门槛：

1. 先读适用的 `AGENTS.md`，再判断任务类型。
2. 划线无法回源、bookId 映射不明、接口失败或资料不完整时，必须保留真实状态，不得猜测、补写或伪造对应材料。
3. 上下文不足时，停在可验证的批次边界，说明已完成范围、待确认材料和下次起点。不要压缩证据或以总论冒充完成。
4. 只读咨询、规则解释和状态检查，不触发文件编辑、Git 操作或任何执行入口调用。
5. 本阶段只改目标中间稿。不要修改正式阅读笔记，不要修改其他书目的材料。

## 2. 中间整理稿与长期笔记的数据边界

中间整理稿是可回源的工作台，应保留 `source ID`、`bookId`、`chapterUid`、`range`、匹配方式、抓取状态和官方版定位等技术字段，供合并、核对、迁移清单和 QA 使用。

中间整理稿是临时工作台，正式阅读笔记才是长期保留文档。

正式阅读笔记是面向阅读的长期文档，只保留划线原文、我自己写的原想法和必要的外部读者原话等可读证据锚点。技术字段不能替代划线原文，不默认写入正文，也不能以标题或编号形式伪装成证据层。

因此，本阶段只写中间整理稿，不写正式阅读笔记，不执行“迁移到正式阅读笔记”。

默认以“我的想法”为主索引整理，优先处理有我的想法的条目；只有划线、没有我的想法的条目暂不整理，除非用户明确要求整理纯划线。

这里的“以我的想法为主索引”，不是只整理我的想法文本。必须同时获取两路数据，用划线接口补足原文、颜色、章节和位置。

如果接口为了定位“本轮新增”需要先拉取全书划线或全部想法，只允许在内部匹配；输出时只展示本轮新增内容，不要把已整理过的旧材料重新输出。

## 3. API 与数据边界

### 3.1 个人数据主接口

1. `/book/bookmarklist`：抓个人划线原文、章节、位置、range、颜色 `colorStyle`。
2. `/review/list/mine`：抓我的个人想法、`abstract`、`chapterUid`、`range`。

### 3.2 官方公开版定位与外部读者接口

1. `/book/bestbookmarks`：抓官方版热门划线，含官方版 range、划线文本和划线人数，用于文本匹配和公开版定位。
2. `/book/underlines`：抓官方版章节内划线热度 range、count、score，只能作为辅助信号，不能单独证明文本同位置。
3. `/book/readreviews`：用官方版 `bookId + chapterUid + range` 抓该划线下其他书友公开想法。
4. `/review/list`：是整本书公开点评 / 公开想法列表，不能按我的章节和 range 定位同一划线位置，不作为同位置高赞想法主接口。

### 3.3 本阶段的接口工作流

1. 先用 `/review/list/mine` 找出本轮新增、且有我自己想法的条目。
2. 再用 `/book/bookmarklist` 补足这些条目的划线原文、章节、位置和颜色。
3. 先在个人笔记坐标内完成匹配，再判断是否需要官方公开版定位。
4. 只有当一条想法已经能稳定合并到划线原文，且能稳定映射到官方公开版同位置时，才默认尝试调用 `/book/readreviews`。
5. 若官方版位置不稳定，记录真实状态，如“未能定位官方版同位置”，不要伪造外部读者材料。

## 4. 官方公开版 bookId 映射规则

本阶段处理任意书目时，都要先分清三件事：个人笔记 bookId、官方公开版 bookId、两者关系。

1. 若个人笔记 bookId 和官方公开版 bookId 相同，可优先使用同一 `bookId + chapterUid + range` 尝试定位。
2. 若个人笔记来自导入书、云书或外部文件，个人划线和想法可能挂在导入版 bookId 上，而公开热门划线和其他书友高赞想法可能挂在官方书城版 bookId 上。
3. 不得因为个人笔记 bookId 下 `/book/bestbookmarks` 为空，或个人 range 直接调用 `/book/readreviews` 返回错误，就判定同位置没有高赞想法。
4. 同位置判断优先按文本内容对应，不按个人笔记所在 `bookId + chapterUid + range` 是否与官方公开版完全一致死判。
5. 若当前书目的官方公开版 bookId 仍未确认，明确记为“待确认”。不要借用其他书的 bookId 补全。
6. 中间稿顶部或本轮说明里，应能看出当前使用的是哪组 bookId 关系：同一 bookId / 导入版与官方版不同 / 待确认。

## 5. 划线与想法匹配规则

不要因为 `bookmarklist` 和 `review/list/mine` 的 `range` 不完全一致，就直接判断为“只有想法”或“无法匹配”。跨段划线、段尾想法、第二段想法都可能导致 `range` 不完全相同。

匹配顺序：

1. **精确匹配**：`bookId + chapterUid + range` 完全一致。
2. **跨段匹配**：`bookId` 和 `chapterUid` 相同，想法 `range` 落在某条跨段划线 `range` 内。
3. **重叠匹配**：`bookId` 和 `chapterUid` 相同，想法 `range` 与划线 `range` 有重叠或相邻，且文本内容连续。
4. **文本判断匹配**：想法内容明显对应某条划线原文，即使 `range` 不完全一致，也可合并，但必须标注“非精确 range 匹配”。

输出时每条都要保留：

1. 匹配方式：精确 range / 跨段 range / 重叠 range / 文本判断匹配 / 未匹配。
2. 是否需要人工确认：是 / 否。
3. 后续处理方向。

效率边界：

1. 不要为了少量未精确匹配条目反复重拉全书、重整旧材料。
2. 未匹配但有想法的条目先列入“有想法但划线匹配待确认”，不要自动扩大整理范围。
3. 如果某条想法没有 `range`，或按以上规则仍无法和划线原文稳定合并，应单独放入“有想法但划线匹配待确认”小节，不要强行归并。

## 6. 同位置其他书友高赞想法默认伴随抓取规则

整理本轮新增内容时，默认以“我写了想法的位置”为索引，尝试查询同一文本位置下其他书友的公开高赞想法。

### 6.1 先决条件

1. 先抓取并合并我的划线、颜色、个人想法。
2. 只根据划线原文和我的原想法，轻量判断为：轻卡 / 完整卡候选 / 主卡候选 / 暂存素材。
3. 轻量预分类只为整理调度和高赞想法抓取深度服务，不等同于第二阶段深度评价。
4. 根据预分类决定同位置高赞想法候选池深度。

### 6.2 官方公开版同位置匹配优先级

1. **精确文本匹配**：我的划线原文和官方版热门划线原文完全一致。
2. **包含匹配**：我的划线原文是官方版划线的一部分，或官方版划线是我的划线的一部分。
3. **高重叠匹配**：两段文字大部分重叠，只是前后多了一两句。
4. **章节内近似文本匹配**：章节标题相同，核心句子相同，但 `range`、段落边界不同。
5. **无法稳定匹配**：找不到足够相同的公开版文本，才标“未能定位官方版同位置”。不能因为我的个人 `range` 没出现在热门划线列表里，就直接判定失败。

### 6.3 候选池与保留数量

1. 所有能稳定定位官方版同位置的条目，默认至少尝试抓取一批 `/book/readreviews` 公开想法，通常 `count=20`。
2. 轻卡默认保留 0 到 1 条；只有当一条外部高赞想法能明显照亮我的触动、形成挑战或补充时才保留。
3. 完整卡候选默认保留 1 到 3 条；如果高赞想法出现多个不同解释方向，应优先保留差异，而不是只保留点赞最高的一条。
4. 主卡候选默认保留 2 到 5 条；必要时可继续翻页扩大候选池，但必须服务于解释分歧、主题张力或问题链推进。
5. 同质重复只保留 1 条代表；方向相反但都贴文本的高赞想法应优先并存。

### 6.4 输出状态必须区分

1. **已获取并保留**：已定位官方版同位置，并筛出有效高赞想法。
2. **已获取但未保留**：接口有外部想法，但多为低质量、同质重复或不能推进文本分析。
3. **未能定位官方版同位置**：个人笔记坐标与官方公开版坐标不同，当前未能通过原文匹配找到对应官方版 range。
4. **同位置暂无公开想法**：已定位官方版同位置并调用 `/book/readreviews`，但 `totalCount` 为 0。
5. **接口错误**：调用失败、鉴权失败、服务端错误等，应记录错误码或简要原因。

高赞想法只能作为外部读者材料，不得替代我的感受、问题和判断。

## 7. 颜色默认流向

微信读书划线颜色通过 `colorStyle` 字段识别。颜色默认流向只作为第一阶段的轻量预分类参考，不是最终卡片类型。

| colorStyle | 颜色名称 | 颜色默认流向 |
|---|---|---|
| `1` | 粉色 / 红色 | 轻卡，记录气氛、心动、暂不深挖的触动 |
| `2` | 紫色 | 人物卡，记录人物心理、关系、命运选择 |
| `3` | 蓝色 | 主题卡，记录阶层、城乡、贫穷、劳动、尊严、现实结构 |
| `4` | 青色 | 细读卡，记录写法、场景、动作、物件、空间、结构、沉默 |
| `5` | 黄色 | 对照卡，记录跨作品、同书前后、作者生平、现实经验联动 |
| `0` | 未分类 | 未分类旧材料 |

输出时不要把颜色写成单纯数字。正文里写“蓝色（主题卡默认流向）”，不要只写 `colorStyle：3`。

整理时必须保持阅读顺序，不要为了按颜色分类而打散阅读线索。颜色统计和主卡候选放在文末汇总即可。

## 8. 想法类别

“想法类别”指我在微信读书里写下的想法本身是什么形态。

1. **感受型**：情绪、心动、惊讶、喜欢、心疼、厌恶、共鸣等，可能很短，但要保留。
2. **浅判断型**：已经有判断，但证据、问题链、文本分析还不充分。
3. **问题型**：主要提出疑问，暂时没有充分判断。
4. **片段分析型**：已经开始分析人物、情节、动作或主题，但还不完整。
5. **完整分析型**：已经接近正式阅读笔记里的完整写法，包含“触动点 / 为什么停下来 / 问题 / 判断 / 扩写方向”等要素，或虽然没显式分栏但分析已经比较完整。
6. **素材型**：主要是为以后文章保存证据、句子、对照材料，当前不急着分析。

## 9. 卡片判断

“卡片判断”指这条材料在后续流程里怎样处理。第一阶段只做轻量预分类，不做深度优化。

1. **轻卡**：保留触动、感受、气氛和直觉，不强行深挖。
2. **完整卡候选**：有明显人物心理、动作细节、结构反常、主题张力，可以补成完整阅读卡。
3. **主卡候选**：不仅能补成完整卡，而且可能发展成文章核心、章节核心或全书母题。
4. **暂存素材**：目前主要作为证据、对照材料或主题材料保存，暂不做完整评价。

轻卡和完整卡不是按字数机械区分，而是看它是否已经形成可追问的矛盾和证据链。

## 12. 第一阶段 QA

执行完成后，至少逐项核对以下问题：

1. 本轮是否只输出本轮新增内容，没有把旧材料重新吐出。
2. 每条已入稿材料是否同时保留了划线原文、我的原想法、颜色、匹配方式、人工确认状态。
3. `颜色默认流向`、`想法类别`、`卡片判断` 是否都只是轻量预分类，没有提前做第二阶段深度优化。
4. 同位置其他书友高赞想法是否只在可稳定定位时抓取；不能稳定定位时，是否明确写出真实状态。
5. 阅读顺序是否保持，没有为了颜色或热度打乱顺序。
6. 本轮索引是否包含轻卡清单、完整卡候选、主卡候选、暂存素材、同位置其他书友高赞想法候选、颜色统计、`下次接着整理位置`。
7. 如果文件原本存在，是否只追加新一轮，而没有重写旧轮次。
8. 是否只修改了目标中间稿，没有改写正式阅读笔记，没有改动其他阅读材料。
9. 是否没有进入第二阶段深度评价，没有执行 `AI修正`，没有写“迁移到正式阅读笔记”的动作。
10. 若本轮仍有“有想法但划线匹配待确认”或“官方公开版待确认”，是否明确写出待确认点和下次起点。

若以上任一项未通过，本阶段不得视为完成。
