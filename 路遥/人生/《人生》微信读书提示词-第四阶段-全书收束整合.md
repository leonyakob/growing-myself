## 内部整合台账

第四阶段开始前，先建立只供内部使用的 `内部整合台账`。它不进入 `/home/king/github/growing-myself/路遥/人生/《人生》阅读笔记.md` 正文。

内部台账至少记录以下多轴字段：

| 字段 | 必填内容 | 说明 |
|---|---|---|
| `source status` | 下方枚举之一 | 先判断材料当前处于什么状态 |
| `archive disposition` | 下方枚举之一 | 再判断这张卡最终怎么归档 |
| `evidence role` | `主题证据` / `人物线证据` / `写法证据` / `阅读轨迹证据` | 说明它承担什么证据作用 |
| `article links` | 0 条、1 条或多条 | 只记录可引用方向，不接管卡片本体 |
| `trajectory flags` | `早期误读` / `后文修正` / `判断变化` | 只给真正有阅读轨迹价值的材料 |
| `external-reader relation` | `回声` / `补充` / `挑战` / `反向解释` / `独立精彩` / `丢弃并说明理由` | 外部读者材料和用户卡之间的关系 |
| `source relation` | `exact same quote` / `overlapping quote` / `same scene` / `different quote` | 处理重复前先定来源关系 |
| `interpretation relation` | `same insight` / `corrected judgment` / `different angle` / `external echo` / `external challenge` | 再定解释关系 |

`source status` 允许值：

1. `已在正式笔记中，结构合格，只补文章索引`
2. `已在正式笔记中，但需补 readable evidence / 用户原句 / 外部原话`
3. `中间稿已优化，尚未迁移`
4. `中间稿与正式稿重复，需合并或建立修订链`
5. `正式稿旧判断已被后文修正，保留为阅读轨迹`
6. `仅外部读者材料，无用户卡片锚点`
7. `来源不足，待回看`

`archive disposition` 允许值：

1. `保留为轻卡`
2. `保留为完整卡`
3. `提升为核心卡`
4. `合并入其他卡`
5. `归档不迁移`
6. `待回看`

判断顺序必须是：先定 `source status`，再定 `archive disposition`，再补 `evidence role`、`article links`、`trajectory flags` 和 `external-reader relation`。不要一上来就把材料塞进某个文章主题。

以下字段只允许留在 `内部整合台账`，不得进入读者可见的正式稿正文或文章 anchors：`source ID`、`range`、`chapterUid`、`bookId`、status fields、interface coordinates、重复决策、QA state。

## 一卡双归宿

第四阶段执行 `一卡双归宿`：每张有价值的卡片都保留自己的归档身份，同时可以被 0 个、1 个或多个文章方向引用。

必须守住四条：

1. 卡片的归档身份优先指它仍然是轻卡、完整卡、核心卡、阅读轨迹卡、暂存证据卡中的哪一种。
2. 一张轻卡即使被 `文章素材索引` 引用，仍然可以保持轻卡，不自动升级为完整卡。
3. 一张完整卡进入 `文章素材索引` 时，仍保留自己的 `划线原文`、`我自己写的内容`、AI评价、AI修正、AI补充等卡片结构。
4. `article links` 只是导航层，不是让文章方向接管卡片本体。`一卡双归宿` 的意思是“档案身份保留，文章链接另挂”，不是“文章标签替代卡片”。

同一张核心卡可以服务多个方向。例如它可以同时进入 `高加林人物线` 和 `知识分子困境`，但不要把两条线压成一条总论。

## 重复卡、改判链与误读修正

第四阶段遇到重复和改判时，先分来源关系，再分解释关系。

来源关系只用四类：

1. `exact same quote`
2. `overlapping quote`
3. `same scene`
4. `different quote`

解释关系只用五类：

1. `same insight`
2. `corrected judgment`
3. `different angle`
4. `external echo`
5. `external challenge`

处理规则：

1. 同一引文、同一判断，多半合并成代表卡，另一条只留在 `内部整合台账` 记重复来源。
2. 同一引文，但判断后来变了，不要合并成一句圆滑结论，而要建立修订链。
3. 同一场景、不同 `source ID` 或不同 `range`，先看是否其实是同一证据场景，不能因为技术坐标不同就算两张独立卡。
4. 不同引文但解释方向不同，可以并存，让它们在 `人物线`、`城乡主题` 或 `写法线索` 里各自承担功能。
5. 外部读者材料若只是回声，可贴在原卡之下。若形成挑战或反向解释，可进入修订链或 `阅读轨迹与判断变化`，但不能改写成我的判断。

当某条早期判断被全书后文修正时，按以下结构收束：

```markdown
- 当时的读法：
- 后来出现的证据：
- 全书后的修正：
- 误读的价值：
- 文章索引用法：
```

这里的“文章索引用法”要区分：

1. 普通主题文默认引用修正后的卡。
2. 只有当文章本身讨论“我的阅读如何变化”时，才把误读链作为 `阅读轨迹证据` 明写出来。

## 外部读者材料与正式稿膨胀防控

第四阶段处理同位置其他书友高赞想法时，继续守住外部读者边界：

1. 外部读者原话只能作为外部读者材料，不能改写成我的第一人称判断，也不能写成“我其实一直觉得”。
2. 只有在有用户卡片锚点时，外部想法才优先随卡内联。若是 external-only 的强材料，先记入 `内部整合台账`，或暂放 `待回看 / 归档不迁移`，不要直接变成正式稿主证据。
3. 同质重复的外部读者想法只保留代表。方向相反但都贴文本的外部读者想法可以并存，用来显示读者分歧。
4. 鸡汤化、纯感叹、离开文本太远、只会重复原文的外部想法，可标成 `丢弃并说明理由`，不要为了显得热闹塞进正文。
5. 正式稿里的文章方向锚点必须是“卡片标题 / 章节或场景 / 引文短句”这类可读锚点，不能写成 `ID 003`、`source ID 17`、`range 233-245`。
6. 正式稿正文不能出现整行 `source status`、`archive disposition`、`QA state`、重复决策或接口坐标。那些都只留在 `内部整合台账`。
7. 如果某张卡已经在正式稿中结构合格，只需补文章索引，不要为了“全书收束”再次复制一遍，避免正式稿膨胀。
8. 如果某张轻卡只能承担气氛证据、语言风格证据、主题回声或文章开头材料，就让它继续做轻卡。不要因为它能入索引，就强行升级。

| Case 9, final-note bloat prevention / archive-discard-with-reason | 对低信息量弱卡执行 `归档不迁移`，必要时只在 `待回看 / 归档不迁移` 里写短原因 | 为了显得材料多，把弱卡也塞进正文或把内部判重理由整行贴进读者可见部分 |

至少逐项自检：

1. 是否保住了 `划线原文`、`我自己写的内容`、`我的原想法` 和必要的外部原话。
2. 是否把 `阅读现场档案` 放在 `文章素材索引` 之前。
3. 是否让 `文章素材索引` 只引用卡片，不替代卡片，不复制整张卡。
4. 是否没有把 `source ID`、`range`、`chapterUid`、`bookId` 写成正式稿正文或文章方向锚点。
5. 是否没有把轻卡一律升级成完整卡。
6. 是否没有让 `AI修正` 覆盖 `我的原想法` 或 `我自己写的内容`。
7. 是否没有把 external-only 材料偷渡成用户主证据。
8. 是否没有把同场景不同坐标误算成两张独立卡。
9. 是否没有为了全书收束再次复制已经结构合格的卡。
