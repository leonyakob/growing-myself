## 3. 多轮整理规则

《人生》可能会分多轮整理。中间整理稿不按轮次新建多个文件，而是持续追加到同一个 `《人生》中间整理稿.md`。

规则：

1. 第 N 轮第一阶段，只追加第 N 轮新增内容，不重写前几轮内容。
2. 第 N 轮第二阶段，默认只优化第 N 轮新增内容，不重新优化整个中间稿。
3. 第 N 轮第三阶段，默认优先迁移第 N 轮已优化完成、适合沉淀的内容；同时检查前几轮已经优化但尚未迁移到正式阅读笔记的遗留内容。若这些遗留内容仍有价值，应一并分类安置；若暂时不迁移，应在中间稿中标明“待迁移 / 待回看”，避免长期遗漏。
4. 只有当我明确说“回看全书中间稿”“整合前几轮”“重整全部内容”时，才跨轮重写或整体重排。
5. 每轮结束都要给出“下次接着整理位置”，方便下一轮继续。

## 6. 大任务处理策略

如果本轮内容体系庞大，不要为了完成任务牺牲分析深度。复杂任务只改变执行组织方式，不得遗漏已纳入范围的用户材料，也不能把该深挖的卡片压成概括、润色或流程填空。

### 6.1 正常任务

符合以下情况时，可以直接执行：

- 处理少量且彼此纠缠度低的完整卡 / 主卡候选；
- 或主要处理轻卡、暂存素材；
- 或完整卡 / 主卡虽有一定分量，但主题线比较集中、文件边界清楚；
- 或只是抓取、合并、轻量优化。

正常任务仍必须遵守文学分析质量规则。不能因为数量少，就只做润色、概括或格式填充。

### 6.2 复杂任务

符合以下任一情况时，必须先制定计划，再分步执行：

- 某张核心卡已经牵涉多条人物线、主题线、后文回看或跨作品联动；
- 多张完整卡 / 主卡候选彼此勾连，需要协调顺序、分批或回看点；
- 涉及多个目标文件、多个阶段产出或正式阅读笔记结构调整；
- 人物关系、城乡主题、尊严主题、爱情线、知识分子困境等多条线索交织；
- 需要把多轮中间稿整合成章节化、主题化的正式笔记；
- 多卡优化、批量迁移、提示词修改、跨多条卡片的质量修复，或任何可能影响既有结构的多文件调整；
- 用户明确强调不要牺牲深度和质量，或当前任务虽不算多但牵一发而动全身。

复杂任务必须先制定详细计划。计划至少列明：目标、范围、禁止事项、输入材料、输出结构、分步任务、目标文件与小节、源材料编号、分批顺序、每批质量门、迁移前预检、迁移后 QA、已知语义回归哨兵，以及回滚或复核点。计划草案先交 Momus、Metis、Oracle 审看，把审查意见吸收到计划里，再把已审计划提交给我确认。计划获确认后，仍须等我明确授权开始执行，才能通过当前客户端实际注册的 start-work 入口进入执行阶段。我的使用环境里可能写作 `/start-work <plan-name>`，某些 skill 文档里也可能写作 `$start-work <plan-name>`，但前缀并不通用，不能把裸 `start-work`、`/shared/start-work` 或 `/start work` 当成统一命令。只写完计划、只得到计划确认，或只听到“start work”这样的自然语言意图，都不等于可以直接展开复杂任务。它们只能视为执行意图，还要落实到当前运行环境的实际入口。

#### 6.2.1 计划工件 canonical block

以下规则与 `.omo/plans/sediment-fifth-stage-prompts-initial-review.md` 的 5.2 canonical block 保持同文，避免《人生》router 另写一份语义漂移版本：

> **计划工件默认规则**：当当前任务已经触发复杂任务计划门、阶段计划门、提示词修改计划门，或用户 / 当前适用提示词明确要求“建 plan / 起草计划 / 制定详细计划 / 先过计划门”时，AI 应直接创建或更新 `.omo/plans/` 下的计划工件，不需要额外再问一次“是否允许写 `.omo/plans`”。这属于计划阶段本身，不等于执行产品文件修改。
>
> **上位禁止覆盖规则**：如果用户当轮明确要求“不写任何文件 / 不要修改任何文件 / read-only / do not edit files / 只在聊天里给计划 / 不落盘计划 / 不创建计划文件 / 不写 `.omo`”，则不得创建或更新 `.omo/plans/` 或 `.omo/drafts/`；除非用户同时明确豁免“允许写计划工件”。如果用户只是说“不要执行 / 不要改产品文件 / plan-only / 先给计划”，这不等于禁止 `.omo/plans` 计划工件；在当前任务触发计划门时，仍可直接创建或更新 `.omo/plans`。
>
> **`.omo/drafts` 限制规则**：`.omo/drafts/` 不是每次建 plan 的必需工件；只有当用户明确要求、当前 workflow / skill 明确要求 durable draft，或计划阶段确实需要保存非最终工作草稿时，才创建或更新 `.omo/drafts/`。`.omo/drafts/` 只能作为非产品 durable draft / 工作草稿，不得被当作绕过审批的产品文件修改位置。
>
> **非执行边界**：`.omo/plans` / `.omo/drafts` 可以保存计划、审查意见、拟议片段、验收标准和验证清单，但必须标明其为 planning / draft artifact。写入这些工件不代表已经获得修改提示词、阅读笔记、中间稿、既有成果文件、文章草稿或其他用户可见产品文件的执行授权；也不允许据此 apply patch 到产品文件、替换目标提示词、更新阅读笔记、stage/commit Git，或把草稿内容当成已批准产品内容。
>
> **变更报告规则**：执行后或交付前报告变更范围时，应把文件分成两类：`计划/草稿工件`（如 `.omo/plans/`，必要时 `.omo/drafts/`）与 `产品目标文件`。计划工件的存在不等于产品文件获批执行；产品文件变更仍只能限于用户明确批准的目标文件。

复杂任务的执行原则：

1. 先判断任务范围和最需要深挖的卡片。
2. 再按主卡候选、完整卡候选、轻卡、暂存素材，或按章节分批处理。
3. 每一批都必须逐条回到划线原文、我的原想法和具体文学分析。
4. 分批不是删减，计划不是替代细读。
5. 不得为了上下文长度、赶速度、统一格式或版面美观，牺牲文本证据、我的原意、分析深度、卡片层次或语义准确性。
6. 只改变执行组织方式，不遗漏已纳入范围的用户材料，也不把该深挖的卡片压平。

### 6.3 正式稿完成后的止损规则

当正式稿已经满足材料完整、书籍原文保留、我的原想法保留、必要外部读者原话保留、结构基本成立且关键误读已修复后，后续只能做有明确目标的定点校补，不再整体重写或为了统一腔调重排全书。

如果我要求恢复某一节的旧风格，优先从 Git 历史中恢复对应范围，再做必要的局部校正。不要重新生成整节来替代原有内容。

## 8. 《人生》已知回归风险与定点校勘样本

这些样本用于复杂任务的计划、迁移后 QA 和最后抽读。它们是防止语义回归的哨兵，不是要求把无关材料硬写进每一轮笔记。

1. **ID 003，马占胜“留意 / 很快妥善解决”**：AI修正不能只把“留意”和“妥善解决”写成漂亮判断，必须补出前史，说明这套说法在既有处境中为何值得警惕。
2. **ID 006，高加林不愿掏炭**：不能简单归因于他当过教师。要比较劳动在他眼中的位置、身份想象与尊严焦虑，避免用单一履历替代心理过程。
3. **ID 021，田晓霞联想**：必须回应我对田晓霞的联想，不能偷换成少平的故事或用另一个人物代替原问题。
4. **ID 109，县城变小和陌生感**：不能把后文失势倒灌到当前场景。此处只能依据眼前的县城、关系和心理变化判断，后文只能作为待回看。
5. **ID 117，“他是农民时爱农民，干部时爱广播员”**：不要因为句子过于整齐就批评它。应保留这句的文气，再补出使判断成立的文本证据桥梁。
6. **刘玉海救灾处**：文本能支持的是救灾队伍迟到五小时，不能擅自改写成救灾队伍在路上耽搁五小时。
7. **黄亚萍的物质付出**：不能简单说这不是牺牲。要承认其中真实的物质成本，同时区分它与巧珍在匮乏中的托举，避免把不同处境的付出抹平。
