# 微信读书五阶段提示词系统设计说明

本文档是微信读书五阶段提示词系统的内部设计说明，用于解释系统定位、权威层级、文件结构、阶段边界、维护方式、QA 机制与扩展规则。它是维护文档，不是运行时最高规则。若本文档与当前用户指令、`AGENTS.md`、书目子目录 `AGENTS.md` 或当前适用提示词冲突，执行时以前者为准，并把冲突作为维护问题处理。

## 1. 系统定位

这套系统不是自动笔记整理器，也不是把微信读书材料快速压成结论的流水线。它服务的是长期文学阅读、思考和写作训练，目标是先保住阅读现场，再逐步推进分析、归档、全书收束与写作转化。

系统的根定位来自项目的“作家驱动的主题化阅读”方法：

1. 用作家保证深度。
2. 用主题训练思想。
3. 用作品提供证据。
4. 用背景增加纵深。
5. 用对照防止狭窄。
6. 用写作完成转化。

这意味着系统追求的不是“笔记更整齐”，而是下面四件事同时成立：

- 用户原始反应不被过早整理掉。
- 文气保留，但必须回到文本证据。
- 阶段任务彼此分开，避免一种认知动作挤压另一种认知动作。
- 最终能把整本书的材料压成可写、可辩、可回证的写作问题。

## 2. 核心原则

下面原则既是写作原则，也是维护和 QA 的判断句。

1. **文本优先**：判断必须能回到原文、情节、动作、场景或明确材料来源，不能拿抽象概念代替文本落点。
2. **用户原始感受优先保留**：短句、火气、疑问、停顿感先保住，再谈优化，不得先替用户说得更完整。
3. **文气必须有文本落点**：好句子可以留，但不能悬空，必须能指出它在回应哪一处文本。
4. **提问可以跳远，论证不能偷步**：问题可以大胆，但回答和判断必须靠证据铺路。
5. **轻卡不强行深挖**：轻卡可以只记录气氛、心动、刺痛感，不必每次都升级成主题判断。
6. **AI 修正不能覆盖用户原文**：正式归档时，用户原话必须保留独立位置，AI 只能补、评、修，不能偷换原文身份。
7. **外部读者材料不能伪装成用户判断**：同位置其他书友高赞想法只能作为外部读者材料，不能改写成“用户本来就这样想”。
8. **文章索引不能吞掉卡片**：第四阶段可以建立文章素材索引，但不能拿索引替代卡片档案，更不能删除阅读现场的复杂性。
9. **第五阶段不回退成整理**：第五阶段负责问题压缩、反证修正和写作转化，不负责重跑卡片整理或重新归档。
10. **通用层与书目专用层不得互相污染**：通用层只写通用机制，专用层只写本书事实和本书回归样本。
11. **保护性规则只能变严，不能被下层文件放宽**：涉及 no-write、执行授权、Git 授权、用户原文、外部读者身份的规则，只能加严，不能降级。
12. **设计说明只解释，不夺权**：本文档负责解释和维护，不直接覆盖运行时规则。

## 3. 文件结构与权威层级总览

### 3.1 文件 / 目录角色总览

| 文件 / 目录 | 作用 | 工件类型 | 是否产品文件 | 维护注意 |
|---|---|---|---|---|
| `AGENTS.md` | 项目全局定位、文学训练规则、微信读书处理边界、文件与 Git 总边界 | 规则文件 | 否 | 是所有后续层的上位约束，不得被提示词或设计说明削弱。 |
| `作者/书名/AGENTS.md` | 某书或某书目子目录的局部规则、专属约束、归档路径说明 | 规则文件 | 否 | 只在当前书任务中生效，不得把 sibling book 规则带进来。不存在时不得凭空假定内容。 |
| `微信读书通用提示词.md` | 通用 router，总入口，负责阶段路由、通用 QA、计划门、新书生成规则 | 规则文件 | 否 | 是通用层首要入口，也是 `.omo/plans` canonical block 的首要维护源。 |
| `微信读书通用提示词-第一阶段-生成中间整理稿.md` 到 `微信读书通用提示词-第五阶段-全书回看与写作转化.md` | 五个通用阶段的执行正文 | 规则文件 | 否 | 只写本阶段认知动作、输入输出、禁止事项和 QA，不写具体书目事实。 |
| `作者/书名/《某书》微信读书提示词.md` | 书目专用 router，固定路径、bookId、书目事实、回归哨兵与专用入口 | 规则文件 | 否 | 可特化本书，但不得把本书经验倒灌回通用层。 |
| `作者/书名/《某书》微信读书提示词-第一阶段...第五阶段...md` | 某书各阶段的专用执行正文 | 规则文件 | 否 | 只在通用阶段边界内写本书专属内容，不能改写通用阶段定义。 |
| `作者/书名/中间整理稿*.md` | 本轮新增材料的临时工作台 | 产品文件 | 是 | 允许整理、归并、轻量预分类，不是长期归档，不替代正式阅读笔记。 |
| `作者/书名/<书名>阅读笔记.md` | 正式阅读笔记，长期档案 | 产品文件 | 是 | 归档采用四段式，必须保留用户原文身份。 |
| `作者/书名/*第四阶段*.md` | 全书收束成果，保存阅读现场档案和文章素材索引 | 产品文件 | 是 | 负责全书结构整合，不负责直接写终稿。 |
| `作者/书名/*第五阶段*.md` | 全书回看与写作转化成果 | 产品文件 | 是 | 负责问题压缩、反证修正、写作路径，不替代最终文章成文。 |
| `.omo/plans/` | 复杂任务计划工件目录 | 计划工件 | 否 | 可在触发计划门且无上位 no-write 禁令时创建，但不等于产品文件执行授权。 |
| `.omo/drafts/` | durable 草稿目录，仅在必要时使用 | 草稿工件 | 否 | 不是默认必需目录，只有用户要求或 workflow 明确需要时才使用。 |
| `.omo/evidence/` | 证据工件目录，必要时保存回源截图、核对记录或其他辅助证据 | 证据工件 | 否 | 只做证据留痕，不替代正式产品文件。 |
| `.omo/run-continuation` | 运行续跑工件，服务任务续跑和状态恢复 | 运行工件 | 否 | 不得把它当产品授权或设计说明正文来源，也不应在普通写作任务中随意改动。 |
| 书目专用 router 中的回归哨兵段落 | 某书回归样本的默认容器 | QA / 证据工件 | 否 | 默认存放处，只有阶段专属错误才下沉到对应书目阶段文件。 |

### 3.2 权威层级与冲突处理

| 层级 | 角色 | 冲突处理 |
|---|---|---|
| 当前用户指令 | 当轮任务的最高约束，尤其是 no-write、执行授权、Git 授权 | 不得被任何下层文档削弱。 |
| 根 `AGENTS.md` | 项目全局定位、文学训练规则、文件与 Git 总边界 | 所有提示词和本文档都不得违背。 |
| 书目子目录 `AGENTS.md` | 当前书的局部规则、专属路径、专属边界 | 执行该书任务时与根规则共同读取，不得导入其他书规则。 |
| 通用 router | 五阶段路由、通用计划门、通用 QA、新书生成规则 | 负责通用层，不得写入本书事实。 |
| 通用阶段文件 | 各阶段通用执行模型 | 不得削弱通用 router 与根规则。 |
| 书目专用 router | 固定路径、bookId、书目事实、书目回归哨兵、专用入口 | 只能特化本书，不能改写通用层定义。 |
| 书目专用阶段文件 | 某书某阶段的执行正文 | 只能在通用阶段边界内做本书特化。 |
| 本设计说明 | 解释性、维护性文档 | 不直接覆盖运行时规则。若发现冲突，先按上层规则执行，再把冲突登记为维护问题。 |

冲突处理总原则：凡是涉及用户原文、文本证据、外部读者身份、no-write、执行授权、Git 授权的规则，只能变严，不能被低层文件放宽。

## 4. 五阶段工作流

| 阶段 | 核心认知动作 | 输入 | 输出 | AI 权限 | 禁止事项 | QA |
|---|---|---|---|---|---|---|
| 第一阶段：生成中间整理稿 | 保存阅读现场，做轻量预分类和批次整理 | 本轮新增划线、想法、颜色、章节位置、外部读者候选材料 | 中间整理稿 | 保存权、合并权、轻量预分类权 | 不得深评，不得写正式阅读笔记，不得把外部读者材料改写成用户判断 | 检查原文是否保留，颜色和位置是否未漂移，本轮新增内容是否完整，游标是否更新 |
| 第二阶段：优化中间整理稿 | 推进细读，升级问题链、判断和张力 | 第一阶段中间整理稿、必要回源文本 | 优化后的中间整理稿 | 分析权、问题升级权、判断修正权 | 不得正式归档，不得覆盖用户原想法，不得把轻卡硬拔成主卡 | 检查问题链是否贴文本，判断是否有证据桥，AI 补充是否仍可追溯 |
| 第三阶段：迁移到正式阅读笔记 | 把稳定材料归档成长期档案 | 经第二阶段处理后的材料、回源核对结果 | 正式阅读笔记条目 | 归档权、回源预检权、来源身份固定权 | 不得覆盖用户原文，不得跳过四段式，不得做整本书收束 | 检查四段式是否完整，用户原文与 AI 内容是否分层，外部材料身份是否清楚 |
| 第四阶段：全书收束整合 | 进行全书结构整合，建立阅读现场档案和文章素材索引 | 正式阅读笔记、书中关键卡片、回源文本 | 第四阶段全书收束成果 | 全书结构收束权、文章素材导航权 | 不得吞掉卡片，不得把索引写成文章，不得替代第五阶段写作转化 | 检查卡片档案是否还在，索引是否只导航不夺权，整合是否覆盖全书而不扁平化 |
| 第五阶段：全书回看与写作转化 | 压缩核心问题，处理反证，生成写作路径 | 第四阶段成果、正式阅读笔记、必要回源文本 | 第五阶段写作转化成果 | 核心问题压缩权、反证修正权、写作转化权 | 不得重跑整理，不得补做第三阶段归档，不得直接替代最终文章成文 | 检查七模块是否齐全，反证是否真的修正判断，写作方向是否指出最容易写坏的地方 |

五阶段的核心不是“先后顺序”本身，而是认知权限拆分。维护者若发现规则越写越厚，优先判断是不是阶段边界不清，而不是继续补局部禁令。

## 5. 通用层与书目专用层边界

| 维度 | 通用层负责 | 书目专用层负责 | 禁止写法 |
|---|---|---|---|
| 系统目标 | 五阶段通用方法、认知权限、通用边界 | 本书怎样落地这些通用方法 | 不得把某一本书的结论写成通用目标。 |
| 路由 | 通用阶段入口、通用任务分流、计划门 | 本书阶段入口、本书固定路径、本书专用说明 | 通用 router 不得固定具体书路径。 |
| 文件路径 | 路径规范、命名模式 | 当前作者目录、书名目录、实际文件路径 | 不得让新书借用旧书路径。 |
| bookId | 只规定“需要确认个人版与官方公开版关系” | 本书个人 bookId、官方公开版 bookId、待确认状态 | 通用层不得写任何具体书的 bookId。 |
| 书目事实 | 不负责具体情节、人物、锚点 | 本书人物、事件、原文锚点、书目事实 | 不得把《人生》的三问、人物、九个锚点写进通用层。 |
| 回归样本 | 只保留 schema、机制、维护规则 | 本书回归哨兵、本书阶段特定错误样本 | 不得复制别的书回归样本当新书样本。 |
| 第四、第五阶段特化 | 只定义第四、第五阶段通用模块和禁止事项 | 本书文章方向、本书三问、本书写作路径、本书阶段哨兵 | 专用层不得反过来重定义阶段本身。 |
| 新书接入 | 通用生成规则与校验清单 | 具体新书的路径、bookId、占位符替换结果 | 不得沿用旧书残留人物、判断或材料。 |

边界判断规则：

1. 只要内容离开当前书仍成立，它就更像通用层内容。
2. 只要内容一换书就失效，它就应留在书目专用层。
3. 只要内容涉及具体人物、bookId、原文锚点、回归样本，它默认属于书目专用层。
4. 书目专用层可以比通用层更具体，但不能比通用层更宽松。
5. 发现通用层与专用层互污时，先把错误内容移回所属层，再补回归样本防止复发。

## 6. 阅读现场保护机制

### 6.1 材料身份保护

微信读书材料进入系统后，先看“这是什么材料”，再看“该怎么处理”。

1. 划线，保存原文锚点。
2. 想法，保存第一反应。
3. 颜色，保存用户粗分类。
4. 同位置其他书友高赞想法，保存为外部读者材料。
5. 中间整理稿，是工作台。
6. 正式阅读笔记，是长期档案。

任何一步都不得把“材料身份”提前抹平。最危险的错误不是句子写得差，而是把不同来源的材料写成同一种声音。

### 6.2 批次保护，`本轮新增内容` 与 `下次接着整理位置`

阅读现场不只靠内容保护，也靠批次边界保护。系统必须显式记录两个东西：

- **本轮新增内容**：从上次整理停止点到当前阅读停顿点之间，新增的划线、想法与相关材料。
- **下次接着整理位置 / 游标**：本轮整理结束后，明确下次从哪里继续，防止重复整理、遗漏材料或批次边界漂移。

维护要求：

1. 每次整理都应说明本轮覆盖的范围。
2. 每次整理结束都应给出下次接着整理位置。
3. 若游标不清楚，优先补游标，不要继续追加分析。
4. 不得把“本轮新增内容”机械等同于自然日、自然周或自然月。

### 6.3 正式归档四段式

| 归档段 | 作用 | 保护点 |
|---|---|---|
| 我自己写的内容 | 完整保留用户原文表达 | 只允许极少量错别字修正，不改写成 AI 口吻。 |
| AI评价 | 评价原文好在哪里、问题在哪里 | 说明判断依据，不抢占原文位置。 |
| AI修正 | 在不改换原意的前提下做更准确的优化表达 | 不能覆盖原文，不能偷换立场。 |
| AI补充 | 补另一个角度、问题链、扩写方向、跨作品联动 | 只能补充，不能倒灌成“用户原本就这样想”。 |

### 6.4 卡片层级保护

1. **轻卡**：记录气氛、心动、暂不深挖的触动，不强行升级。
2. **完整卡**：已经有感受、问题、判断、扩写中的若干部分，可以系统评价。
3. **主卡 / 核心卡**：承担书中关键问题、证据链和文章方向。
4. **细节卡**：从具体细节往抽象走，先把动作、物件、空间和沉默写清楚。
5. **主题卡**：从抽象命题往具体情境落，先追问这句话发生在什么现实矛盾中。

保护结论：第四阶段不能吞掉这些层级，第五阶段也不能跳回去重洗它们。

## 7. 写作转化机制

### 7.1 第五阶段进入条件

只有下面条件同时成立，才能进入第五阶段：

- 整本书已经读完。
- 第四阶段已经完成。
- 已有阅读现场档案和文章素材索引。
- 用户明确要求进入第五阶段。

若不满足这些条件，优先回到正确阶段，不用第五阶段代替缺失步骤。

### 7.2 第五阶段七个标准模块

| 标准模块 | 作用 | 必做检查 |
|---|---|---|
| 核心问题压缩 | 把整本书材料压成可写、可争论、可展开的问题 | 不是主题词，不是口号，必须能回到文本矛盾。 |
| 证据-反证表 | 汇总支持判断的证据与抵触判断的反证 | 必须包含“反证如何修正判断”字段，反证不是装饰。 |
| 张力地图 | 标出相反力量的碰撞点和消解点 | 必须回到文本，不得空谈张力。 |
| 五异法细读点 | 从逻辑反常、情感反常、字词突兀、结构断裂、刻意沉默中筛出细读入口 | 不硬凑，不为了凑满五类而牵强附会。 |
| 写作转化 | 把材料转成可执行的写作路径 | 必须写出“最容易写坏的地方”。 |
| 作家写法档案 | 记录本书写法观察，并区分稳定观察与待验证假设 | 不能把一次观察直接当作作家通则。 |
| 跨作品联动 | 给出比较路径，推进原问题而不是只堆书名 | 必须说明联动后问题如何深化。 |

### 7.3 `证据-反证表` 的最低字段

`证据-反证表` 至少应包含下列字段：

1. 初步判断。
2. 支持证据。
3. 反证或抵触材料。
4. **反证如何修正判断**。
5. 修正后的判断。
6. 仍待补证的空白。

### 7.4 第五阶段不得做什么

1. 不得把第五阶段写成主题总结。
2. 不得拿第五阶段补第三阶段归档。
3. 不得把第五阶段当成第四阶段的索引扩写。
4. 不得直接把第五阶段成果冒充最终文章终稿。

## 8. QA 与回归样本

QA 在本系统里是语义保护，不是格式检查。它要保护三件事，文本没有漂，材料身份没有乱，任务授权没有越界。

### 8.1 三层 QA

| QA 层级 | 防护目标 | 典型错误 | 典型检查 |
|---|---|---|---|
| 文本层 | 防事实、措辞、引文意义漂移 | 把“晚到了五个小时”改成“在路上耽误五个小时” | 回源核对、引文意义核对、动作和主语核对 |
| 材料层 | 防用户原话、AI 补充、外部读者想法混淆 | 把高赞想法改写成用户自己的判断 | 四段式归档检查、材料来源身份检查、颜色与游标核对 |
| 任务层 | 防阶段误判、计划变执行、复杂任务越权修改 | 把 `.omo/plans` 当成产品文件执行授权 | 阶段判断、计划门检查、执行授权检查、 Git 授权检查 |

### 8.2 回归样本字段

| 字段 | 必填要求 | 说明 |
|---|---|---|
| 触发场景 | 必填 | 说明错误发生在什么任务、什么阶段、什么材料环境下。 |
| 源证据 | 必填 | 指向原文、原笔记、截图、定位信息或其他可回源材料。 |
| 错误类型 | 必填 | 例如事实漂移、材料身份混淆、阶段越权、外部材料倒灌。 |
| 禁止重犯处理方式 | 必填 | 写清这类错误以后不能怎么写、不能怎么改。 |
| 预期处理 | 必填 | 说明正确处理应该保住什么、补什么、限制什么。 |
| 通过条件 | 必填 | 说明怎样才算修复成功。 |
| review / update 状态 | 必填 | 标记有效、待复核、合并、失效等状态。 |

### 8.3 回归样本存放位置与生命周期

1. 书目回归哨兵默认放在书目专用 router。
2. 只有当某个错误只保护特定阶段行为时，才放入对应书目阶段文件。
3. 通用层只保留回归样本 schema、维护规则和检查机制，不保留具体书目样本。
4. 必要时可把辅助证据留存在 `.omo/evidence/`，但证据工件不替代书目 router 中的样本规则。
5. 回归样本只从真实错误中沉淀，不虚构样本，不预支样本。
6. 样本不是越多越好。过期、重复、无法继续提供保护的样本，要复核、合并或标记失效。

## 9. 复杂任务计划门与 `.omo/plans` 规则

### 9.1 何时触发计划门

遇到下面情况之一，应先进入计划门：

1. 任务跨多个模块或多个阶段。
2. 任务要求修改提示词体系或维护系统规则。
3. 任务本身不清楚，需要先做决策性拆解。
4. 当前适用提示词明确要求先建 plan。
5. 用户明确要求“先做计划”。

### 9.2 计划最小内容

计划至少要写清：

- 目标。
- 范围。
- 输入。
- 输出。
- 风险。
- 目标文件。
- 验收标准。
- 禁止事项。

### 9.3 三方审查职责

| 审查者 | 主要职责 | 关注重点 |
|---|---|---|
| Metis | 查隐藏意图和真实需求 | 有没有误读用户目标，有没有把内部维护文档写成宣传稿或散文。 |
| Momus | 查计划结构、遗漏和可执行性 | 是否有缺章、缺验证、缺边界，后续执行者能不能直接照计划写。 |
| Oracle | 查架构风险和长期维护风险 | 权威层级是否清楚，通用 / 专用边界是否稳定，长期是否易漂移。 |

### 9.4 `.omo/plans`、`.omo/drafts` 与执行授权

1. 当任务触发计划门，且没有更高层的 no-write、read-only、“不写 `.omo`”禁令时，可以直接创建或更新 `.omo/plans` 下的计划工件。
2. 用户批准计划，只代表认可方案，不等于产品文件执行授权。只有用户明确说“批准执行”“开始执行”或同等意思时，才进入产品文件修改。
3. 若用户明确说“不写任何文件”“只在聊天里给计划”“read-only”“不落盘计划”“不写 `.omo`”，则不得写 `.omo/plans` 或 `.omo/drafts`，除非用户明确豁免。
4. `.omo/drafts` 不是默认必需目录，只在用户要求或 workflow 明确需要 durable draft 时使用。
5. 计划工件和草稿工件，都不等于产品文件修改授权。
6. 变更报告必须区分计划 / 草稿工件与产品目标文件，不能混报。
7. 阶段切换、计划通过、文档写完，都不等于 Git 授权。
8. Git 操作必须单独获得授权。

### 9.5 canonical block 的首要维护源与同步

1. `.omo/plans` 相关 canonical block 的首要维护源，是通用 router `微信读书通用提示词.md` 中的复杂任务计划门规则。
2. 通用第五阶段、书目专用 router 或书目阶段文件若复制了该 block，应视为同步文本。
3. 修改 canonical block 时，必须检查所有复制位置，避免规则词面一致但语义漂移。
4. 若专用层只引用通用规则而不复制全文，也要保证继承关系、路径和适用说明清楚。

## 10. 新书接入流程

### 10.1 Preflight

- [ ] 检查作者目录和书名目录是否已存在。
- [ ] 读取根 `AGENTS.md`。
- [ ] 若书目子目录 `AGENTS.md` 已存在，读取它；若不存在，不凭空假定。
- [ ] 检查是否已有专用 router、五个阶段文件、中间整理稿、正式阅读笔记，未经明确授权不得覆盖。
- [ ] 确认作者目录名、书名、文件命名格式。

### 10.2 接入执行清单

- [ ] 创建书目专用 router。
- [ ] 创建五个书目专用阶段提示词文件。
- [ ] 固定中间整理稿路径。
- [ ] 固定正式阅读笔记路径。
- [ ] 固定专用 router 和五个阶段文件路径。
- [ ] 确认个人版 bookId。
- [ ] 确认官方公开版 bookId。
- [ ] 若 bookId 未知，明确标注“待确认”，不得编造。
- [ ] 第一阶段前确认公开热门划线映射关系。
- [ ] 不预先制造回归样本，等真实错误出现后再沉淀。

### 10.3 Postflight

- [ ] 专用 router 与五个专用阶段文件齐全。
- [ ] 所有作者名、书名、占位路径都已替换完成。
- [ ] 个人版 bookId 和官方公开版 bookId 已确认，或清楚标为待确认。
- [ ] 没有旧书人物、旧书 bookId、旧书回归样本、旧书专属判断残留。
- [ ] 路由 smoke check 能正确区分第一到第五阶段、只咨询、只整理、只评价问题质量、提示词修改。
- [ ] 未创建虚构回归样本。

## 11. 维护与扩展规则

| 维护动作 | 必须做 | 不得做 |
|---|---|---|
| 修改通用 router | 只改通用路由、通用计划门、通用 QA、新书生成规则 | 不得写入任何具体书人物、bookId、原文锚点、三问或回归样本 |
| 修改通用阶段文件 | 只改本阶段通用执行模型、输入输出、禁止事项、QA | 不得夹带某一本书的专属判断 |
| 修改书目专用 router | 维护本书路径、bookId、书目事实、回归哨兵和专用入口 | 不得反向改写通用阶段定义 |
| 修改书目专用阶段文件 | 只改本书当前阶段的特化内容 | 不得把本书经验抽成所有书通用规则 |
| 修改 canonical block | 先确认通用 router 里的首要维护源，再同步所有复制位置 | 不得只改某个复制块而忘记其他位置 |
| 维护回归样本 | 检查样本是否仍有效，必要时合并、复核或标失效 | 不得把虚构错误、过期样本或别的书样本硬塞进当前书 |
| 扩展第五阶段经验 | 先在具体书目中验证，再考虑抽象进通用层 | 不得因为一次成功案例就直接升级为通用规则 |
| 规则变厚时重构 | 先检查是否需要拆阶段、拆文件、拆边界 | 不得只靠继续加禁令补丁 |

维护补充规则：

1. router 只写入口、路径、规则摘要，不复制阶段长正文。
2. 阶段文件只写本阶段执行正文，不承担系统总说明。
3. 若设计说明与实际提示词出现冲突，先按当前适用规则执行，再修正文档，不反过来拿文档压规则。
4. 维护新书时，若书目子目录 `AGENTS.md` 不存在，不要伪造内容，可在后续明确需要时再建立。

## 12. 已知风险与防护

| 风险 | 表现 | 防护方式 |
|---|---|---|
| 提示词过厚 | 单个文件堆满例外和补丁，维护越来越慢 | 优先拆阶段、拆文件、拆边界，不靠无限追加规则 |
| 阶段误判 | 本该做整理却直接做写作转化，或本该归档却继续讨论 | 用五阶段工作流表和进入条件做前置判断 |
| 轻卡被过度加工 | 本来只需保住触动，却被强行升级成宏大主题 | 轻卡单独保护，不把“有感觉”误当“必须论证” |
| 用户原文被 AI 修正覆盖 | 正式阅读笔记里只剩 AI 语言，看不见用户原话 | 严格执行四段式，用户原文独立存放 |
| 外部读者想法混入用户判断 | 高赞想法被写成用户自己的视角 | 保持外部读者材料标签，单独存放身份 |
| 文章素材索引吞掉卡片 | 第四阶段只剩提纲，卡片档案被抹平 | 明确第四阶段产物是“档案 + 索引”，不是索引替代档案 |
| 通用层与专用层互相污染 | 通用层写进本书事实，或专用层反向重定义通用规则 | 用职责对照表、回归样本和评审检查双向防污染 |
| 《人生》经验被误当通用模板 | 新书直接借用《人生》三问、锚点或人物关系 | 把《人生》压到附录，只允许作为案例，不允许作为模板 |
| `.omo/plans` 被误解成执行授权 | 写完计划就直接改产品文件 | 明确计划批准不等于执行授权，产品修改还要等明确批准 |
| 回归样本过期或虚构 | 样本越来越多，却不再提供真实保护 | 只从真实错误沉淀，定期复核、合并或标失效 |
| QA 成本过高 | 每次任务都重做全部检查，效率失衡 | 用三层 QA 定位风险，把检查压到最必要处 |
| Git 提交混入无关文件 | 计划工件、运行工件和产品文件被混在同一提交里 | Git 单独授权后先看 status 和 diff，只处理授权范围 |
| 权威层级不清 | 设计说明、提示词、用户指令互相打架，执行者不知道听谁 | 固定权威层级表，冲突时只向上找约束，不向下找放宽 |
| canonical block 漂移 | 通用层和专用层都写了计划门，但内容慢慢不一致 | 以通用 router 为首要维护源，修改时逐一同步复制位置 |

## 附录 A：《人生》作为案例

《人生》是本系统的重要验证案例，但它只是案例，不是通用模板，也不是本文档的主轴。

### A.1 它能说明什么，不能说明什么

| 维度 | 能说明什么 | 不能说明什么 |
|---|---|---|
| 专用层 | 书目专用 router 如何固定路径、双 bookId 和专用入口 | 不能说明所有书都该沿用同一路径和同一组字段值 |
| 第四阶段 | 如何把整本书材料收成“阅读现场档案 + 文章素材索引” | 不能说明第四阶段应该替所有书生成同一种索引结构 |
| 第五阶段 | 如何把第四阶段材料压成可写问题、反证和写作路径 | 不能说明所有书第五阶段都应使用同一组三问 |
| 回归样本 | 如何用真实错误防止事实漂移、材料身份混淆和阶段退化 | 不能说明别的书可以直接复制《人生》的回归哨兵 |

### A.2 《人生》在专用层中的典型信息

1. 它有书目专用 router 和专用阶段文件。
2. 它固定了个人导入版与官方公开版的双 bookId。
3. 它验证了第四阶段与第五阶段应分别承担“保存复杂性”和“压缩成可写问题”的不同任务。

### A.3 《人生》第五阶段的书目专属结果

《人生》第五阶段最终压出的三问，是本书专属结果，不是其他书的模板：

1. 读书把高加林打开了，为什么还是没给他一个能站稳的位置？
2. 为什么在高加林那里，爱、劳动和体面总要先经过别人的眼睛？
3. 为什么高加林总先行使爱的权利，却迟迟不肯承担爱的责任？

其递进链条是：`安身失败 → 外部目光夺权 → 关系责任后撤 → 巧珍承担代价`。

这一链条只说明第五阶段可以怎样把具体书目的材料压成可写问题，不说明所有书都必须照此提问。

### A.4 使用边界

维护者在引用《人生》时，应遵守下面边界：

1. 可以拿它说明专用层如何工作。
2. 可以拿它说明第四阶段和第五阶段的差异。
3. 可以拿它说明回归样本如何保护语义。
4. 不得把高加林、巧珍、黄亚萍等人物分析写进通用规则。
5. 不得把三问、九个锚点、回归哨兵和双 bookId 写成新书默认模板。
