# 微信读书 Skill 通用整理提示词

用途：当阅读任意一本书时，用这份通用提示词生成该书的专用微信读书整理提示词。专用提示词应固定作者、书名、中间整理稿路径和正式阅读笔记路径，避免每次整理时混用通用规则和具体书目规则。

核心目标：

1. 用微信读书数据完整保留我的阅读现场：划线、想法、颜色、章节、位置。
2. 先生成中间整理稿，再优化分析，最后迁移到正式阅读笔记。
3. 中间整理稿是临时工作台，正式阅读笔记才是长期保留文档。
4. 所有分析以文本为落点，以文学性和分析深度为目标，不为了覆盖数量牺牲质量。

---

## 0. 占位说明与路径规则

- `{作者}`：书籍作者或项目中的作者目录名，例如 `路遥`、`余华`。
- `{书名}`：书名，例如 `人生`、`活着`。
- `{中间稿路径}`：建议为 `/home/king/github/growing-myself/{作者}/{书名}/《{书名}》中间整理稿.md`。
- `{正式笔记路径}`：建议为 `/home/king/github/growing-myself/{作者}/{书名}/《{书名}》阅读笔记.md`。
- `{专用提示词路径}`：建议为 `/home/king/github/growing-myself/{作者}/{书名}/《{书名}》微信读书提示词.md`。

例如《人生》：

- 中间整理稿：`/home/king/github/growing-myself/路遥/人生/《人生》中间整理稿.md`
- 正式阅读笔记：`/home/king/github/growing-myself/路遥/人生/《人生》阅读笔记.md`
- 专用提示词：`/home/king/github/growing-myself/路遥/人生/《人生》微信读书提示词.md`

不要把中间整理稿命名成单纯的 `{书名}.md`，避免和书目文件、正式笔记混淆。

## 2. 三阶段总流程

整理微信读书材料时分三阶段：

1. **第一阶段：生成中间整理稿**
   抓取本轮新增划线和想法，合并、排序、初步分类，写入 `{中间稿路径}`。第一阶段只做整理和初步归类，不做深度评价，不写入正式阅读笔记。
2. **第二阶段：优化中间整理稿**
   在中间整理稿内，对本轮新增内容进行轻卡优化、完整卡优化、主卡候选深挖、暂存素材归类。第二阶段重点是分析质量和文学表达，不为了覆盖所有材料而牺牲深度。
3. **第三阶段：迁移到正式阅读笔记**
   把已经优化完成、适合沉淀的内容分类安置到 `{正式笔记路径}`。正式阅读笔记中不保留中间稿编号、临时流程说明和冗余归档痕迹。

未经明确要求，不要直接把微信读书导出的内容写入正式阅读笔记。

---

## 3. 多轮整理规则

一本书通常会分多轮整理。中间整理稿不按轮次新建多个文件，而是持续追加到同一个 `《{书名}》中间整理稿.md`。

每一轮用独立小节，例如：

```markdown
## 第1轮整理：开头至第六章

## 第2轮整理：第七章至第十章

## 第3轮整理：……
```

规则：

1. 第 N 轮第一阶段，只追加第 N 轮新增内容，不重写前几轮内容。
2. 第 N 轮第二阶段，默认只优化第 N 轮新增内容，不重新优化整个中间稿。
3. 第 N 轮第三阶段，默认优先迁移第 N 轮已优化完成、适合沉淀的内容；同时检查前几轮已经优化但尚未迁移到正式阅读笔记的遗留内容。若这些遗留内容仍有价值，应一并分类安置；若暂时不迁移，应在中间稿中标明“待迁移 / 待回看”，避免长期遗漏。
4. 只有当我明确说“回看全书中间稿”“整合前几轮”“重整全部内容”时，才跨轮重写或整体重排。
5. 每轮结束都要给出“下次接着整理位置”，方便下一轮继续。

## 4. 阶段切换前 Git 检查点

当我要求进入第二阶段时，例如：

```text
现在根据【提示词】进行第二阶段，优化《{书名}》中间整理稿
```

AI 必须先完成 Git 检查点：

1. 检查第一阶段生成的 `{中间稿路径}`。
2. 查看 `git status`，确认本阶段变更。
3. 只提交第一阶段相关文件，不混入无关修改。
4. 推送到 GitHub。
5. 确认 push 成功后，才能开始第二阶段。

当我要求进入第三阶段时，例如：

```text
现在根据【提示词】进行第三阶段，迁移到《{书名}》正式阅读笔记
```

AI 必须先完成 Git 检查点：

1. 检查第二阶段优化后的 `{中间稿路径}`。
2. 查看 `git status`，确认本阶段变更。
3. 只提交第二阶段相关文件，不混入无关修改。
4. 推送到 GitHub。
5. 确认 push 成功后，才能开始第三阶段。

注意：

- 本提示词中的阶段切换提交，属于用户明确要求的阶段性提交。
- 仍需遵守项目 `AGENTS.md` 的 Git 规则。
- 如项目要求 Git 命令加前缀，则使用 `GIT_MASTER=1`。
- 不要提交无关阅读材料、临时文件或其他书目的内容。

## 5. 数据抓取硬约束

微信读书里“划线颜色”和“我的想法”来自不同数据口径：

- 划线原文、章节、位置、颜色：来自 `/book/bookmarklist`，颜色字段是 `colorStyle`。
- 我的个人想法：来自 `/review/list/mine`。

整理单本书时，必须同时获取两路数据，并用：

```text
bookId + chapterUid + range
```

合并同一段原文。

不要只整理我的想法。

每条整理材料至少保留：

- 章节名
- `chapterUid`
- `range`
- 划线颜色
- 划线原文
- 我的原想法
- 想法类别
- 卡片判断
- 后续处理方向

如果接口为了定位“本轮新增”需要先拉取全书划线或全部想法，只允许在内部匹配；输出时只展示本轮新增内容，不要把已整理过的旧材料重新输出。

如果某条想法没有 `range`，无法和划线原文稳定合并，应单独放入“无法定位到划线的想法”小节，不要强行归并。

## 6. 颜色默认流向

微信读书划线颜色通过 `colorStyle` 字段识别。颜色只作为默认流向，不是最终卡片类型。

内部识别规则：

- `colorStyle=1`：粉/红色，轻卡，记录气氛、心动、暂不深挖的触动。
- `colorStyle=2`：紫色，人物卡，记录人物心理、关系、命运选择。
- `colorStyle=3`：蓝色，主题卡，记录阶层、城乡、贫穷、劳动、尊严、现实结构。
- `colorStyle=4`：青色，细读卡，记录写法、场景、动作、物件、空间、结构、沉默。
- `colorStyle=5`：黄色，对照卡，记录跨作品、同书前后、作者生平、现实经验联动。
- `colorStyle=0`：未分类旧材料。

输出时不要把颜色写成单纯数字。正文中写成：

```markdown
- 划线颜色：蓝色（主题卡默认流向）
```

不要写成：

```markdown
- colorStyle：3
```

整理时必须保持阅读顺序，不要为了按颜色分类而打散阅读线索。颜色统计和主卡候选可以放在文末汇总。

## 7. 想法类别与卡片类型

### 7.1 想法类别

“想法类别”指我在微信读书里写下的想法本身是什么形态。

1. **感受型**：情绪、心动、惊讶、喜欢、心疼、厌恶、共鸣等，可能很短，但要保留。
2. **浅判断型**：已经有判断，但证据、问题链、文本分析还不充分。
3. **问题型**：主要提出疑问，暂时没有充分判断。
4. **片段分析型**：已经开始分析人物、情节、动作或主题，但还不完整。
5. **完整分析型**：已经接近正式阅读笔记里的完整写法，包含“触动点 / 为什么停下来 / 问题 / 判断 / 扩写方向”等要素，或虽然没显式分栏但分析已经比较完整。
6. **素材型**：主要是为以后文章保存证据、句子、对照材料，当前不急着分析。

### 7.2 卡片类型

“卡片类型”指这条材料后续怎样处理：

- **轻卡**：保留触动、感受、气氛和直觉，不强行深挖。轻卡不是废料，它记录真实阅读心流。
- **完整卡候选**：有明显人物心理、动作细节、结构反常、主题张力，可以补成完整阅读卡。
- **主卡候选**：不仅能补成完整卡，而且可能发展成文章核心、章节核心或全书母题。
- **暂存素材**：目前主要作为证据、对照材料或主题材料保存，暂不做完整评价。

轻卡和完整卡不是按字数机械区分，而是看它是否已经形成可追问的矛盾和证据链。

## 8. 文学分析质量规则

第二阶段优化时，评价重点放在分析质量、分析深度和文学表达上。AI 的任务不是只判断“对/不对”，而是把我的感受、问题和判断分析透，把我可能还没说出来、但文本中已经潜伏的心理、矛盾和意味表达出来。

### 8.1 必须有文本落点

完整卡和主卡候选的分析必须回到具体文本细节，例如：

- 词语
- 动作
- 表情
- 沉默
- 场景
- 人物关系
- 叙事位置
- 前后情节的变化

不能只用“阶层”“自尊”“爱情”“城乡差异”“命运”等抽象词概括。

### 8.2 人物心理要写出过程，不要直接贴标签

分析人物时，要写清楚心理动作链：

- 人物意识到了什么；
- 他回避了什么；
- 他想维护什么；
- 他害怕承认什么；
- 他在哪一点上发生动摇；
- 他还没有走到哪一步。

不要直接把人物心理写成没铺垫的结论。

例如，不要直接写：

```text
他已经被巧珍的爱打动了。
```

应写出中间层次：

```text
这里更值得分析的不是他是否已经“动情”，而是他已经察觉到巧珍的感情，并本能地回避这种可能。巧珍的真诚先触到的不是爱情，而是他的身份羞耻和自尊防线：他不愿承认自己会被一个“不符合他理想身份”的农村姑娘影响。
```

具体判断仍需根据实际文本调整，不要套用示例。

### 8.3 完整卡和主卡候选必须打开矛盾

每张完整卡或主卡候选至少要指出一个张力点。这里的“张力”不是固定主题清单，而是要求 AI 从当前文本里寻找相反力量的碰撞，例如：

- 人物想要什么 vs 现实不允许什么；
- 人物说出口的话 vs 没说出口的沉默；
- 表面动作 vs 隐藏心理；
- 情感冲动 vs 身份、伦理、习俗或制度限制；
- 自我想象 vs 他人眼光；
- 个人选择 vs 时代、阶层、家庭或命运结构；
- 叙事明写的内容 vs 叙事刻意留白的内容；
- 一个场景里的温情、羞耻、恐惧、欲望、尊严等复杂力量之间的冲突。

这些只是寻找张力的方法，不是要求每本书都套用同一组主题。分析必须从当前作品、当前章节和当前文本细节出发，找到真正存在的矛盾，不要因为示例里出现某类张力就强行套用。

### 8.4 AI 要提供表达增量

AI 不能只说：

```text
这个判断有道理。
这个问题很敏锐。
可以继续发展。
```

必须把用户没说透的地方说出来：

- 补心理层次；
- 补文本细节；
- 补问题链；
- 补文章方向；
- 补一个新的合理视角；
- 尽量用有文气但不空泛的语言表达。

所谓“表达增量”，不是替用户改成学术论文，而是在保留用户真实反应的基础上，把直觉说深，把判断说细，把矛盾说出文学味。

### 8.5 AI 可以反驳，但必须指出缺失的分析环节

如果用户判断跳跃、太宽、太二选一、证据不足或表达不准，AI 应直接指出。

但不要只说：

```text
这不准确。
这里需要修正。
这个判断太跳。
```

而要说明：

- 少了哪一环；
- 哪个文本细节还没接上；
- 当前判断最多能推出什么；
- 如果要推进成更强判断，需要补什么证据或中间分析；
- 可以怎样改成更稳、更深、更有文学性的表达。

### 8.6 大胆分析，但不要省掉过程

可以借助后文回看、作者背景、社会结构、同书内部联动和跨作品比较提出深层解释。

但不能直接空降结论。要写出为什么能这样看、从哪里推进过来、当前文本能支撑到哪一步。

深度不是保守，深度是把判断的来路写出来。

## 10. 第二阶段：优化中间整理稿

```text
请阅读 `{中间稿路径}`，在保持原阅读顺序不变、保留我的原想法不变的前提下，对本轮新增内容做优化整理。

要求：
1. 不要覆盖我的原想法，所有 AI 优化都写在原想法下方。
2. 先做“本轮优化前诊断”，再开始正式优化。
3. 先检查每条的想法类别和卡片判断是否准确；必要时修正。
4. 轻卡按“轻卡优化规则”处理。
5. 完整卡候选和主卡候选按“完整卡 / 主卡优化规则”处理。
6. 暂存素材只说明以后可能服务于哪个主题、人物或对照方向，不强行评价。
7. 不要改写正式阅读笔记。
```

### 10.1 本轮优化前诊断

正式优化前，先写：

```markdown
## 第N轮优化前诊断

- 本轮轻卡数量：
- 本轮完整卡候选数量：
- 本轮主卡候选数量：
- 本轮暂存素材数量：
- 前几轮已优化但尚未迁移的遗留内容：
- 最值得深挖的卡：
- 需要重点展开的人物心理线：
- 可能形成文章核心的问题：
- 建议本次优化范围：
- 是否需要分批：
```

如果本轮内容很多，不要为了完成全部内容而压缩分析。可以按以下方式分批：

1. 先优化主卡候选；
2. 再优化完整卡候选；
3. 再优化轻卡；
4. 最后处理暂存素材归宿；
5. 或按章节分批优化。

如果完整卡候选超过 5 条，或主卡候选超过 2 条，或涉及多条复杂主题线，建议先制定优化计划，再分批执行。

### 10.2 轻卡优化规则

轻卡适用于感受型、浅判断型、短问题、气氛记录。处理目标是保留真实触动，同时把表达稍微修亮，不把它硬拽成完整论文。

每张轻卡按以下结构补在原想法下面：

```markdown
**AI轻评：**

- 这条想法抓住了什么：
- 可以稍微说深的一点：
- 不必强行展开的原因：

**AI优化表达：**

> 
```

要求：

- 先保护真实触动，不强行拔高。
- 如果我的表达太直白，可以优化成更有文学味儿的表达。
- 可以引用相关名言或典故，但必须准确、贴切；不能为了好看硬套，不能编造出处。不确定出处时，不要写成“某某说”。
- 优化表达直接写在原想法下方，不覆盖原想法。
- 不强行补“我提出的问题 / 我的初步判断 / 以后可扩写成什么文章”。

### 10.3 完整卡 / 主卡优化规则

完整卡候选、主卡候选适用于“片段分析型”和“完整分析型”想法，尤其是已经接近正式阅读笔记写法的内容。

处理时必须遵守 `/home/king/github/growing-myself/AGENTS.md` 的读书笔记评价结构，并重点执行“文学分析质量规则”。

每张完整卡或主卡候选按以下结构补在原想法下面：

```markdown
**AI评价：**

- 这条想法抓住的真实问题：
- 好在哪里：
- 目前还没说透的地方：
- 问题质量：
- 判断是否有跳跃：

**AI修正：**

> 

**AI补充：**

- 另一个角度：
- 可以打开的张力：
- 问题链升级：
- 可扩写方向：
- 同书内部联动：
- 跨作品联动：
```

要求：

1. 评价要具体，不能空泛夸“敏锐”。
2. 必须指出文本落点：词语、动作、场景、沉默、人物关系或叙事位置。
3. 必须分析心理过程，不要直接贴人物标签。
4. 必须打开一个张力点。
5. 如果用户判断跳跃，要直接指出缺少哪一环。
6. AI修正必须保留用户原意，不要改成无血肉的说明文。
7. AI补充要提供新的文学分析增量，不要重复用户原话。

### 10.4 暂存素材处理规则

暂存素材不是废料。它可能暂时不适合展开，但以后要有归宿。

每条暂存素材至少说明：

```markdown
**暂存说明：**

- 可归入的主题：
- 可能服务的人物线：
- 以后可用于的文章或对照方向：
- 是否需要等后文回看：
```

## 11. 第三阶段：迁移到正式阅读笔记

只有当我明确说“把已优化内容更新到正式阅读笔记”时，才执行这一阶段。

```text
请把 `{中间稿路径}` 中已经优化完成、适合沉淀的内容，整理并更新到 `{正式笔记路径}`。

要求：
1. 不要机械搬运所有中间稿；要按材料价值分类安置。
2. 默认优先迁移本轮已优化内容，同时检查前几轮已优化但尚未迁移的遗留内容；有价值的应一并迁移或列入待回看，不要默认遗漏。
3. 主卡候选进入重点卡或文章核心候选。
4. 完整卡进入正式阅读卡。
5. 轻卡进入轻卡存档或阅读心流片段。
6. 暂存素材进入主题素材库、人物素材库或对照材料。
7. 暂时无法判断但仍有价值的材料，放入“待回看”小节。
8. 必须保留“我的原想法”，不要用 AI 修正覆盖原文。
9. 按项目要求采用四段式：我自己写的内容 / AI评价 / AI修正 / AI补充。
10. 不要修改其他文件。
```

### 11.1 正式阅读笔记去除中间稿痕迹

正式阅读笔记是长期保留文档，不是过程记录。迁移时禁止保留：

- 中间稿编号；
- 临时编号；
- “本条来自第 N 轮中间稿”；
- 冗长归档说明；
- 多条历史“下次接着整理位置”。

如果确实需要保留整理进度，只在文末保留最新一条：

```markdown
下次接着整理位置：第X章 / chapterUid=xx / range=xxxx-xxxx 之后
```

### 11.2 正式阅读笔记游标更新规则

每次更新正式阅读笔记时，文末只保留一条最新的“下次接着整理位置”。

第 N+1 轮迁移时：

1. 先删除第 N 轮旧的“下次接着整理位置”。
2. 再迁移第 N+1 轮内容，并处理前几轮已优化但尚未迁移的遗留内容。
3. 第 N+1 轮迁移完成后，在文末追加新的“下次接着整理位置”。

不要在正式阅读笔记中累积多条历史游标。

### 11.3 所有材料最终都要有归宿

中间整理稿只是临时工作台。整本书读完后，所有有价值材料都应被分类安置：

- 主卡候选 → 重点卡 / 文章核心候选；
- 完整卡候选 → 完整阅读卡；
- 轻卡 → 轻卡存档 / 阅读心流片段；
- 暂存素材 → 主题素材库 / 人物素材库 / 对照材料；
- 无法定位想法 → 散点想法 / 待回看 / 不迁移。

第三阶段不是只迁移“精华”，而是把不同价值层级的材料放到合适位置。

## 12. 大任务处理策略

如果本轮内容体系庞大，不要为了完成任务牺牲分析深度。

适合直接用 Hephaestus-Deep Agent 的情况：

- 本轮章节不多；
- 完整卡候选较少；
- 主要是抓取、合并、轻量优化；
- 主题线比较集中。

适合先用 Prometheus-Plan Builder 制定计划，再用 `/start work` 执行的情况：

- 已经读了很多章；
- 完整卡候选超过 5 条；
- 主卡候选超过 2 条；
- 人物关系、城乡主题、尊严主题、爱情线、知识分子困境等多条线索交织；
- 需要重整正式阅读笔记结构；
- 需要把多轮中间稿整合成章节化、主题化的正式笔记。

注意：

- Prometheus 适合规划，不替代文本细读。
- 真正优化时仍必须逐条回到划线原文、我的原想法和具体文学分析。
- 如果任务很大，应主动分步骤进行，保证每一步质量。
