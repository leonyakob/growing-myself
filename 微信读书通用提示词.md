用途：当阅读任意一本书时，先用这份总入口判断当前属于哪一阶段，再打开对应阶段文件执行。详细阶段执行正文已经拆到五个阶段文件里；本文件只保留总入口、权威规则、阶段路由、QA / Git 门、复杂任务计划门，以及未来生成书目专用提示词的统一标准。

核心目标：

1. 用微信读书数据完整保留我的阅读现场：划线、想法、颜色、章节、位置，以及同位置其他书友高赞想法。
2. 先生成中间整理稿，再优化分析、迁移、全书收束，最后进入写作转化。
3. 中间整理稿是临时工作台，正式阅读笔记才是长期保留文档。
4. 所有分析以文本为落点，以文学性和分析深度为目标，不为了覆盖数量牺牲质量。
5. 正式阅读笔记必须保留可读证据层：书籍划线原文、我自己写的原想法，以及确有保留价值的外部读者原话。AI评价、AI修正、AI补充只能建立在这些原文之上，不能替代它们。
6. `source ID`、`bookId`、`chapterUid`、`range`、点赞数、抓取状态和官方版定位等技术字段，只服务中间整理、迁移清单和回源核对，不默认进入正式阅读笔记正文。

## 0. 占位说明与路径规则

- `{作者}`：书籍作者或项目中的作者目录名，例如 `路遥`、`余华`。
- `{书名}`：书名，例如 `人生`、`活着`。
- `{中间稿路径}`：建议为 `/home/king/github/growing-myself/{作者}/{书名}/《{书名}》中间整理稿.md`。
- `{正式笔记路径}`：建议为 `/home/king/github/growing-myself/{作者}/{书名}/《{书名}》阅读笔记.md`。
- `{专用路由路径}`：建议为 `/home/king/github/growing-myself/{作者}/{书名}/《{书名}》微信读书提示词.md`。
- `{专用第一阶段提示词路径}`：建议为 `/home/king/github/growing-myself/{作者}/{书名}/《{书名}》微信读书提示词-第一阶段-生成中间整理稿.md`。
- `{专用第二阶段提示词路径}`：建议为 `/home/king/github/growing-myself/{作者}/{书名}/《{书名}》微信读书提示词-第二阶段-优化中间整理稿.md`。
- `{专用第三阶段提示词路径}`：建议为 `/home/king/github/growing-myself/{作者}/{书名}/《{书名}》微信读书提示词-第三阶段-迁移到正式阅读笔记.md`。
- `{专用第四阶段提示词路径}`：建议为 `/home/king/github/growing-myself/{作者}/{书名}/《{书名}》微信读书提示词-第四阶段-全书收束整合.md`。
- `{专用第五阶段提示词路径}`：建议为 `/home/king/github/growing-myself/{作者}/{书名}/《{书名}》微信读书提示词-第五阶段-全书回看与写作转化.md`。
- `{个人笔记bookId}`：我的划线、颜色和个人想法所在 bookId。导入书、云书或外部文件通常会有独立 bookId。
- `{官方公开版bookId}`：微信读书官方书城版或公开热门划线所在 bookId。若暂未确认，在专用提示词中写“待确认”。
- `{bookId关系}`：同一 bookId / 导入版与官方版不同 / 待确认。

过去的单文件 `{专用提示词路径}` 现在只对应书目专用 router，不再承担五阶段执行正文。不要把中间整理稿命名成单纯的 `{书名}.md`，避免和书目文件、正式笔记混淆。

### 0.1 不可降级原则与任务入口门

开始任何整理、优化、迁移、回看或提示词修改前，先执行以下门槛：

1. 先阅读当前任务适用的 `AGENTS.md`，再判断任务类型：第一阶段、第二阶段、第三阶段、第四阶段、第五阶段、只整理、只评价、只咨询、提示词修改或状态检查。
2. 质量、文本证据、用户原意和来源区分，高于速度、上下文经济、格式统一和覆盖数字。
3. 分批只改变执行组织方式，不能遗漏已纳入范围的用户材料，不能删减证据或降低应有的单卡深度。这不等于保留所有外部读者候选，也不等于把所有轻卡升级为完整卡。
4. 划线无法回源、bookId 映射不明、接口失败或资料不完整时，必须保留真实状态，不得猜测、补写或伪造对应材料。
5. 上下文不足时，停在可验证的批次边界，说明已完成范围、待确认材料和下次起点。不要压缩证据或以总论冒充完成。
6. 复杂任务先过计划门：先起草计划，再交给 Momus、Metis、Oracle 审查，吸收审查意见后把审查版计划呈现给用户。只有用户明确授权执行，才进入执行阶段。
7. 用户平时不需要手动切换到 Prometheus 或 Atlas；若任务进入计划流或执行流，可由 Hephaestus 按当前运行时、当前客户端已注册的入口接续。
8. 只读咨询、规则解释和状态检查，不触发文件编辑、Git 操作或任何执行入口调用。

## 1. 如何用通用提示词生成某本书专用提示词

```text

要求：
1. 把所有 `{作者}` 替换成实际作者目录名。
2. 把所有 `{书名}` 替换成实际书名。
3. 固定中间整理稿路径为 `/home/king/github/growing-myself/{作者}/{书名}/《{书名}》中间整理稿.md`。

5. 生成以下六个文件，而不是旧式单文件大提示词：
   - `/home/king/github/growing-myself/{作者}/{书名}/《{书名}》微信读书提示词.md`
   - `/home/king/github/growing-myself/{作者}/{书名}/《{书名}》微信读书提示词-第一阶段-生成中间整理稿.md`
   - `/home/king/github/growing-myself/{作者}/{书名}/《{书名}》微信读书提示词-第二阶段-优化中间整理稿.md`
   - `/home/king/github/growing-myself/{作者}/{书名}/《{书名}》微信读书提示词-第三阶段-迁移到正式阅读笔记.md`
   - `/home/king/github/growing-myself/{作者}/{书名}/《{书名}》微信读书提示词-第四阶段-全书收束整合.md`
   - `/home/king/github/growing-myself/{作者}/{书名}/《{书名}》微信读书提示词-第五阶段-全书回看与写作转化.md`
6. 书目专用 router 负责固定路径、阶段入口、书目级事实、QA / Git / 计划门和该书自己的回归样本沉淀。
7. 五个书目专用阶段文件分别承接第一阶段到第五阶段的执行正文，不再回到旧式单文件格式。
8. 专用提示词里不要再保留通用占位符。
9. 生成内容要能直接复制使用。
10. 生成专用提示词时，应确认该书是否存在“个人笔记 bookId”和“官方公开版 bookId”不同的情况；若已知，应在专用 router 中固定写出两个 bookId；若未知，应保留“待确认”字段，并要求第一轮整理前先确认公开热门划线使用的官方版 bookId。
```

### 专用提示词生成边界与验收门

1. 只迁移可跨书复用的方法、流程门和质量标准，不迁移其他书目的具体人物、主题结论、bookId、源材料编号、历史误读或回归样本。
2. 书目专属事实未经核实，一律标为“待确认”，不能借用其他书的资料补全。
3. 初次生成专用提示词不得虚构回归样本。只有该书出现真实、可定位的问题后，才可在该书专用 router 中沉淀相应的回归哨兵。
4. 生成后检查：书目专用 router 与五个书目专用阶段文件齐全，路径明确，未知事实已标明“待确认”，且没有混入其他书目的专属内容。

## 2. 五阶段总流程

详细执行正文已经移动到对应阶段文件。本节只保留总入口路由、阶段边界和进入 / 退出条件摘要。

1. **第一阶段：生成中间整理稿**
   - 打开 [微信读书通用提示词-第一阶段-生成中间整理稿.md](./微信读书通用提示词-第一阶段-生成中间整理稿.md)。
   - 抓取本轮新增划线、想法和同位置其他书友高赞想法，合并、排序、轻量预分类，写入 `{中间稿路径}`。
   - 第一阶段只做整理、轻量预分类和高赞想法初步筛选，不做深度评价，不写入正式阅读笔记。
2. **第二阶段：优化中间整理稿**
   - 打开 [微信读书通用提示词-第二阶段-优化中间整理稿.md](./微信读书通用提示词-第二阶段-优化中间整理稿.md)。
   - 在中间整理稿内，对本轮新增内容进行轻卡优化、完整卡优化、主卡候选深挖、暂存素材归类。
   - 第二阶段重点是分析质量和文学表达，不为了覆盖所有材料而牺牲深度。
3. **第三阶段：迁移到正式阅读笔记**
   - 打开 [微信读书通用提示词-第三阶段-迁移到正式阅读笔记.md](./微信读书通用提示词-第三阶段-迁移到正式阅读笔记.md)。
   - 把已经优化完成、适合沉淀的内容分类安置到 `{正式笔记路径}`。
   - 正式阅读笔记中不保留中间稿编号、临时流程说明和冗余归档痕迹。
4. **第四阶段：读完整本书后的全书收束整合**
   - 打开 [微信读书通用提示词-第四阶段-全书收束整合.md](./微信读书通用提示词-第四阶段-全书收束整合.md)。
   - 在整本书读完、第三阶段已有基础后，以 **阅读现场档案 + 文章素材索引** 对全书材料做一次整书级收束。
   - `阅读现场档案` 开头要先交代本书各阅读轮次、阶段或批次的问题推进，再列卡片入口；全书完成后要把“下次接着整理位置”切换为完成态，并清理 `ledger`、`snapshot`、`task`、`wave` 等内部流程词。
   - 第四阶段重点是补齐全书结构、保留阅读轨迹、建立文章方向导航，不把文章素材索引写成文章草稿，也不把第三阶段全部重跑一遍。
5. **第五阶段：全书回看与写作转化**
   - 打开 [微信读书通用提示词-第五阶段-全书回看与写作转化.md](./微信读书通用提示词-第五阶段-全书回看与写作转化.md)。
   - 在整本书读完且第四阶段已经完成后，把全书材料继续压缩为可写、可反驳、可推进的核心问题结构。
   - 第五阶段重点是核心问题收束、证据与反证对齐、张力主次压缩和写作转化，不回退成补卡整理，也不把总入口写成第五阶段长正文。

未经明确要求，不要直接把微信读书导出的内容写入正式阅读笔记。

## 3. 多轮整理规则

一本书通常会分多轮整理。中间整理稿不按轮次新建多个文件，而是持续追加到同一个 `《{书名}》中间整理稿.md`。

每一轮用独立小节，例如：

```markdown
## 第1轮整理：开头至第六章

## 第2轮整理：第七章至第十章

## 第3轮整理：……
```

规则：

1. 第 N 轮第一阶段，只追加第 N 轮新增内容，不重写前几轮内容。
2. 第 N 轮第二阶段，默认只优化第 N 轮新增内容，不重新优化整个中间稿。
3. 第 N 轮第三阶段，默认优先迁移第 N 轮已优化完成、适合沉淀的内容；同时检查前几轮已经优化但尚未迁移到正式阅读笔记的遗留内容。若这些遗留内容仍有价值，应一并分类安置；若暂时不迁移，应在中间稿中标明“待迁移 / 待回看”，避免长期遗漏。
4. 只有当我明确说“回看全书中间稿”“整合前几轮”“重整全部内容”时，才跨轮重写或整体重排。
5. 每轮结束都要给出“下次接着整理位置”，方便下一轮继续。

## 4. 阶段切换前 QA 与条件式 Git 检查点

当我要求进入第二阶段时，例如：

```text
现在根据【提示词】进行第二阶段，优化《{书名}》中间整理稿
```

AI 必须先完成第一阶段的 QA：

1. 检查第一阶段生成的 `{中间稿路径}`。
2. 核对本轮范围、划线原文、我的原想法、匹配待确认项和下次接着整理位置。
3. 确认第一阶段没有深度评价，也没有改写正式阅读笔记。

当我要求进入第三阶段时，例如：

```text
现在根据【提示词】进行第三阶段，迁移到《{书名}》正式阅读笔记
```

AI 必须先完成第二阶段的 QA：

1. 检查第二阶段优化后的 `{中间稿路径}`。
2. 核对本轮优化范围、卡片类型处理、原想法保留和核心卡质量门。
3. 确认第二阶段没有改写正式阅读笔记。

当我要求进入第四阶段时，先确认整本书已经读完，第三阶段已有基础，再按 `微信读书通用提示词-第四阶段-全书收束整合.md` 中的“fixture-based QA”和“完成标准”执行。

注意：

- QA 是阶段切换的必要条件，阶段切换本身不构成 commit 或 push 授权。
- 只有当前用户指令明确要求 commit、push 或其他 Git 操作时，才执行相应操作。当前用户指令优先于本提示词的一般规则。
- 获得明确 Git 授权后，先查看 `git status`，只处理本阶段相关文件，不混入无关阅读材料、临时文件或其他书目的内容。
- 仍需遵守项目 `AGENTS.md` 的 Git 规则。如项目要求 Git 命令加前缀，则使用 `GIT_MASTER=1`。

## 5. 复杂任务处理策略

如果本轮内容体系庞大，不要为了完成任务牺牲分析深度。复杂任务只改变执行组织方式，不遗漏已纳入本轮范围的用户材料，也不降低单卡深度。

### 5.1 正常任务

符合以下情况时，可以直接执行：

- 处理 1-2 张完整卡 / 主卡候选；
- 或主要处理轻卡、暂存素材；
- 或完整卡 / 主卡数量虽不多，且主题线比较集中；
- 或只是抓取、合并、轻量优化。

正常任务仍必须遵守“文学分析质量规则”。不能因为数量少，就只做润色、概括或格式填充。

### 5.2 复杂任务

符合以下任一情况时，必须先制定计划，再分步执行：

- 至少 1 张核心卡牵涉多条人物线、主题线、叙事结构、后文回看或跨作品联动；
- 涉及多个目标文件、多个阶段产出或正式阅读笔记结构调整；
- 人物线、主题线、叙事结构、现实背景、写法分析等多条线索交织；
- 需要把多轮中间稿整合成章节化、主题化的正式笔记；
- 用户明确强调不要牺牲深度和质量。

复杂任务的执行原则：

1. 先判断任务范围和最需要深挖的卡片。
2. 再按主卡候选、完整卡候选、轻卡、暂存素材，或按章节分批处理。
3. 每一批都必须逐条回到划线原文、我的原想法和具体文学分析。
4. 分批不是删减；计划不是替代细读。
5. 不遗漏已纳入本轮范围的用户材料；外部读者候选仍按价值筛选，不以数量保留为目标；不降低单卡深度，只改变执行组织方式。

复杂任务进入计划门时，计划工件统一遵守以下 canonical block：

> **计划工件默认规则**：当当前任务已经触发复杂任务计划门、阶段计划门、提示词修改计划门，或用户 / 当前适用提示词明确要求“建 plan / 起草计划 / 制定详细计划 / 先过计划门”时，AI 应直接创建或更新 `.omo/plans/` 下的计划工件，不需要额外再问一次“是否允许写 `.omo/plans`”。这属于计划阶段本身，不等于执行产品文件修改。
>
> **上位禁止覆盖规则**：如果用户当轮明确要求“不写任何文件 / 不要修改任何文件 / read-only / do not edit files / 只在聊天里给计划 / 不落盘计划 / 不创建计划文件 / 不写 `.omo`”，则不得创建或更新 `.omo/plans/` 或 `.omo/drafts/`；除非用户同时明确豁免“允许写计划工件”。如果用户只是说“不要执行 / 不要改产品文件 / plan-only / 先给计划”，这不等于禁止 `.omo/plans` 计划工件；在当前任务触发计划门时，仍可直接创建或更新 `.omo/plans`。
>
> **`.omo/drafts` 限制规则**：`.omo/drafts/` 不是每次建 plan 的必需工件；只有当用户明确要求、当前 workflow / skill 明确要求 durable draft，或计划阶段确实需要保存非最终工作草稿时，才创建或更新 `.omo/drafts/`。`.omo/drafts/` 只能作为非产品 durable draft / 工作草稿，不得被当作绕过审批的产品文件修改位置。
>
> **非执行边界**：`.omo/plans` / `.omo/drafts` 可以保存计划、审查意见、拟议片段、验收标准和验证清单，但必须标明其为 planning / draft artifact。写入这些工件不代表已经获得修改提示词、阅读笔记、中间稿、既有成果文件、文章草稿或其他用户可见产品文件的执行授权；也不允许据此 apply patch 到产品文件、替换目标提示词、更新阅读笔记、stage/commit Git，或把草稿内容当成已批准产品内容。
>
> **变更报告规则**：执行后或交付前报告变更范围时，应把文件分成两类：`计划/草稿工件`（如 `.omo/plans/`，必要时 `.omo/drafts/`）与 `产品目标文件`。计划工件的存在不等于产品文件获批执行；产品文件变更仍只能限于用户明确批准的目标文件。

复杂任务的计划、审查和执行授权是三个独立的门：

1. 先起草计划，写清任务范围、分批策略、质量门、涉及文件和预期产出。
2. 起草后必须交给 Momus、Metis、Oracle 审查，不能跳过其中任何一方。
3. 必须把三方审查意见吸收进计划，补齐范围、风险、QA 和执行边界，再向用户呈现审查后的计划。
4. 用户批准计划，只代表认可这份审查后的方案，不等于已经明确授权执行。
5. 用户平时不需要手动切换到 Prometheus 或 Atlas；若当前运行时由 Hephaestus 负责路由，应由它把计划流和执行流接到对应工作流。
6. “开始执行计划”或 `start work` 可以视为自然语言执行意图，但仍必须映射到 currently registered start-work entry for this client。
7. 若当前环境文档给出 `/start-work <plan-name>` 或 `$start-work <plan-name>`，它们都只是环境示例，不能被宣称为所有客户端的唯一通用写法。
8. 不要把 `/shared/start-work`、`裸 start-work`、`/start work` 或裸写 `start-work` 当成面向用户的通用执行命令；若当前环境没有可用的已注册 start-work 入口，必须暂停并说明，不能静默直接执行。
9. 用户明确说“审查补全后执行”时，可视为用户提前给出“审查完成后再执行”的意图，但仍必须先完成 Momus、Metis、Oracle 审查、吸收审查意见，并在满足当前客户端入口规则后再执行。

### 5.3 止损与定点校补

正式稿已达标后，只做定点校补：围绕用户指出的卡片、证据、结构或表述修正，不借机重写整章或全书。只有用户明确要求全书重整时，才重新制定计划，按本节的复杂任务规则完成审查，确认范围并取得明确执行授权。

## 7. 专用提示词回归样本沉淀规则

每本书只在出现真实、可定位的 AI 误读、漏读、过度概括、磨平用户文气或迁移错位风险后，才在该书的专用 router 中沉淀“回归样本”或回归哨兵。

回归样本应写明：触发场景、错误类型、必须保留或核对的证据、禁止重犯的处理方式和可验证的通过条件。它服务于该书的后续整理与 QA，不应被复制到通用提示词或其他书目的专用提示词。

通用提示词不虚构任何书目的回归样本。初次生成专用提示词时，只保留本节的机制，不填入具体案例。
