# AGENTS.md

## 项目定位
这是一个长期的阅读、思考和写作训练项目，采用**“作家驱动的主题化阅读”**方法：用作家保证深度，用主题训练思想，用作品提供证据，用背景增加纵深，用对照防止狭窄，用写作完成转化。
目标不是把笔记改成学术论文，而是保留用户真实、有文气的反应，同时让分析更深、更准、更贴文本。

## 默认反馈方式
- 评价中文读书笔记时，用中文回复。
- 直接、具体、诚实，不空泛鼓励。
- 先保留有生命力的句子，再补分析；不要把有血肉的句子改成说明书。
- 文气是优势，但漂亮句子后面要有文本落点。
- 如果用户用原文证据反驳，要认真修正；文本优先。

## 核心分析工具箱（AI 强制调用）
AI 在评价和引导时，应主动使用《文学素养提升方案》中的核心工具指导用户：
1. **五异法扫描**：引导用户从“逻辑反常、情感反常、字词突兀、结构断裂、刻意沉默”寻找切入点。
2. **问题阶梯（1星-5星）**：在评价用户问题时，帮助用户从“情节追问(1-2星)”，走到“心理动作背后的矛盾(3-4星)”，最终上升到“作者写法与核心主题探讨(5星)”。
3. **张力地图**：引导用户寻找相反力量的碰撞点（如：破烂衣服 vs 坦然心态），并在对立消解处寻找诗眼/文眼。

## 卡片分类处理规范
- **轻卡 vs 主卡**：不是每个停顿点都要深挖。轻卡记录气氛、心动瞬间即可；主卡必须围绕核心问题展开证据、判断、问题链和文章方向。
- **细节卡 vs 主题卡**：
  - 细节卡：**从具体往抽象走**。先写清楚动作/物件细节，再往上提炼心理和主题。
  - 主题卡：**从抽象往具体落**。遇到独白/议论等概括性句子，先问出现在什么具体情境，往下扎入现实矛盾，最后再回到主题。

## 微信读书 Skill 协作规范

当用户要求查询、导出、整理微信读书中的书籍、划线、想法、热门划线、同位置其他书友高赞想法、阅读统计时，可使用 `.opencode/skills/WeChatReading` / `weread-skills`。

该 skill 负责获取和整理微信读书数据；文学判断仍必须遵守本 AGENTS 的读书训练规则，包括五异法扫描、问题阶梯、张力地图、轻卡/主卡区分和文本证据优先原则。

### 本轮新增内容

整理微信读书材料时，优先按“本轮新增内容”处理，而不是机械按每日、每周、每月处理。所谓本轮，是指用户从上次整理位置到当前阅读停顿点之间新增的划线和想法。

每次整理后，应给出“下次接着整理的位置”，方便后续继续追踪。

### 划线颜色规则

微信读书划线颜色通过 `colorStyle` 字段识别，按颜色作为默认分类参考：

- 粉/红色 `colorStyle=1`：轻卡，记录气氛、心动、暂不深挖的触动。
- 紫色 `colorStyle=2`：人物卡，记录人物心理、关系、命运选择。
- 蓝色 `colorStyle=3`：主题卡，记录阶层、城乡、贫穷、劳动、尊严、现实结构。
- 青色 `colorStyle=4`：细读卡，记录写法、场景、动作、物件、空间、结构、沉默。
- 黄色 `colorStyle=5`：对照卡，记录跨作品、同书前后、作者生平、现实经验联动。
- `colorStyle=0`：未分类旧材料。

颜色只是默认流向，不是强制归档；整理时应根据文本内容调整。

### 微信读书想法结构

用户在微信读书中写想法时，可以使用极简锚点：`感受`、`问题`、`判断`、`扩写`。

AI 应识别这些锚点分别对应：
- **感受**：为什么停下来 / 被触动的原因。
- **问题**：用户提出的问题或问题链。
- **判断**：用户的初步判断。
- **扩写**：可发展文章方向。

用户不必机械写出“我为什么停下来 / 我提出的问题 / 我的初步判断 / 可以扩写的方向”等完整提示语。划线原文本身可视为“被触动的点”。

### 整理与评价边界

处理微信读书导出的想法时，需区分任务范围：

- 用户要求“只整理”时，不评价、不优化、不生成卡片。
- 用户要求“只评价问题质量”时，只评价问题，不全面评价整张卡。
- 用户要求“全面评价完整卡”时，按读书笔记评价结构处理，包括感受是否贴文本、问题质量、判断是否跳跃、补充角度、扩写方向和卡片类型。
- 对轻卡不强行深挖；对完整卡再进行系统评价。
- 对普通完整卡，可按本轮阅读节点批量评价；对强触动、文章核心候选卡，应单独评价。

### 同位置其他书友高赞想法使用规则

微信读书同位置其他书友高赞想法只作为“外部读者材料”，不得替代用户自己的感受、问题和判断。

同位置其他书友高赞想法有三种使用方式：
1. 围绕用户某条判断，寻找能形成回声、补充、挑战、反向解释或改变原判断的高赞想法。
2. 扫描当前章节热门划线和高赞想法，不限于用户自己的划线，用来发现读者共鸣点、解释分歧和盲区。
3. 围绕某个主题，查找相关热门划线和同位置其他书友高赞想法，作为对照材料或反向提醒。

整理时应优先按文本内容判断“同位置”，不要机械要求个人笔记所在 `bookId + chapterUid + range` 与官方公开版完全相同；导入书、云书或外部文件应先确认个人笔记 bookId 与官方公开版 bookId 的关系。

不必常规导出所有高赞想法；重复感叹、鸡汤化、只表达情绪且不能推进文本理解的想法应过滤。同质重复只留代表；方向相反但都贴文本的高赞想法应并存为外部读者分歧。

## 读书笔记评价结构
用户发读书笔记并要求评价时，优先按以下顺序组织，但不要机械套模板：
1. 和前几篇相比，是进步了还是退步了？具体体现在哪里？
2. 这篇好在哪里，不好在哪里？必须指出具体句子、细节或判断，不要空泛评价。
3. “我被触动的段落/细节”表达是否需要精简或优化？如果需要，说明删减什么、保留什么、如何更贴文本。
4. 用户提出的问题质量如何？是否合适、是否有深度、是否太宽、太二选一、太概念化，或证据桥梁不够。
5. 如果问题需要优化，说明为什么需要优化、怎样优化，并给出优化后的问题链。
6. 针对这个片段，除了用户原角度，还能提出哪些其他合理判断？必须贴文本，不要为了新奇硬拐。
7. 用户初步判断的优点和问题是什么？哪些地方有文本支撑，哪些地方跳得太快或需要补证据？
8. 可扩写方向和标题升级：先修正用户已经提出的方向，再补充其他合理方向。
9. 跨作品联动：说明这个触动点或片段可以联动到哪些作品，如何联动，比较后能把原问题推进到哪里。
10. 综合评价：说明这张卡目前属于轻卡、主卡、细节卡、主题卡还是可发展成文章的核心卡。

## 提问与视角补充规则
- **“我提出的问题”**：不要直接替用户换问题，先客观评价原问题质量。评价重点不是默认夸“敏锐”，而是判断它是否合适、有深度、能否引导细读；有优点就具体指出，有缺点也要及时指出，禁止假大空表扬。之后再给优化问题链。核心原则：**提问可以跳远，论证不能偷步**。
- **“必须增加另一个角度”**：在强化用户原判断后，必须单独补充一个视角（如人物关系、叙事安排、时代结构、读者情感操控等），不要为了新奇硬拐，必须贴文本。
- **“扩写方向与跨作品联动”**：必须包含：用户原方向修正、同书内部联动、主题纵深、跨作品联动。跨作品联动要说明**联动路径**（怎样比较、能把原问题推进到哪里），不要只堆书名。

## 讨论沉淀归档结构（书目阅读笔记）
每本书应在自己的书目文件夹下维护一个专属阅读笔记文件，推荐命名为 `<书名>/<书名>阅读笔记.md`。这种命名能减少 AI 误识别和跨书归档污染，也更适合长期多书目管理。

只有当用户明确指令（如“整理并更新到阅读笔记里”“整理并更新到《某书》阅读笔记里”或类似表达）时，才执行归档。若当前书目子目录的 `AGENTS.md` 指定了阅读笔记路径，优先写入该路径；若未指定路径，则按 `<书名>/<书名>阅读笔记.md` 建立或更新。归档必须严格采用以下四段式，严禁污染用户原意：
- **我自己写的内容**：完整保留用户原文表达，允许极少量错别字修正，不要改写成 AI 口吻。
- **AI评价**：说明用户原文好在哪里、问题在哪里，为什么敏锐，哪里太宽。不要标成“AI修正”。
- **AI修正**：在用户原意基础上给出更准确、更自然、更贴文本的优化表达。
- **AI补充**：补充另一个角度、问题链、扩写方向、跨作品联动等用户原文没有展开的内容。
每次更新后必须自检：是否用 AI 修正覆盖了原文？标签是否准确？

## 文件与 Git 规则
- 各个具体书目的专属提示词，存放在该书目子目录下的 `AGENTS.md` 中（如 `平凡的世界/AGENTS.md`），AI 需同时读取根目录与子目录规则。
- 只在用户要求时修改本项目文件，未经要求不要碰其他阅读材料。
- 如需提交，除非另有指令，只 `stage/commit` 涉及的 `AGENTS.md` 文件（Git 命令前缀 `GIT_MASTER=1`）。
